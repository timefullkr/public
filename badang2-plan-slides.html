바당 — 그릇과 알맹이
왜 지금인가, 바당(LMS)이 비워 둔 다섯 가지, 그리고 그 공백을 채울 계층의 국내외 위치.
§0 개발 배경 — 왜 지금인가

학생은 이미 AI를 쓰고 있다

  • 공교육의 책임 — 공교육은 시대에 맞는 교육 인프라를 직접 조성해, 모든 학생이 그 시대의 핵심 역량을 차별 없이 갖추도록 책임져야 한다.
  • 지금 그 역량은 AI 활용이다 — 그리고 학생은 이미 쓰고 있다. 금지가 아니라 안전하고 형평하게 보장하는 것이 과제다.전체 논의는 「AI 교육 패러다임」
  • 그래서 필요한 것 — 실시간 수업 · AI 접근 · 안전 · 기록을 한데 갖춘 공공 수업 플랫폼. 바당2가 그 인프라다.
§0 바당과 바당2의 관계

재구축이 아니라, 그릇 위에 얹는 ‘AI 계층’이다

  • 바당은 그대로 둔다 — 소스 복제·분기(fork)가 아니라, 이미 갖춰진 개방형 표준에 밖에서 끼우는(plug-in) 외부 도구.NIA·타 시·도와 코드 충돌·라이선스 문제 없음
  • 제주 전용 · 직접 소유 — 11개 시·도 공동(다자 합의로 경직·지연)과 달리, AI 변화에 제주가 기민하게 확장·변경.
  • 바당에는 AI가 없다 / 바당2가 그 AI다 — 바당은 학습기록 정본 저장소(LRS), 바당2는 송수신 전문 원본 확보 + 안전·거버넌스.
  • 수업을 넘어 업무까지 — 학생·교사 수업 + 교직원 행정을 동일 게이트웨이로.단, 세션 유형으로 권한·안전·감사 정책 분리(§3-6)
포크(복제)가 아니라 표준 연계 — 바당 코드는 건드리지 않는다
§1 바당(AIEP) 현황 — 다섯 공백

그릇은 완성됐다 — 다섯 가지가 비어 있다

①
자체 생성형 AI 기능이 없다 — 구축 현안 문서가 명시: “AI모델 부재(2차 구축 시 개발 필요)”. 학생이 직접 쓸 생성형 AI 대화·튜터가 없다.
②
학습데이터 수집에 한계가 있다 — “빅테크 외 연계 에듀테크 학습데이터 수집 불가”. SSO로 연결만 할 뿐, 송수신 전문을 교육청이 확보하지 못한다.
③
분석·활용이 미흡하다 — 분석 고도화는 2차 구축 과제. 데이터가 흩어져 효과 검증·생기부 근거로 쓰기 어렵다.
④
안전·거버넌스가 공백이다 — 교육청이 직접 안전필터·로깅·통제하는 장치가 없다.
⑤
사업 중복·혼선이 우려된다 — AIEP(11시도/NIA) ↔ 교육부(KERIS) 트랙 간 예산 중복·혼란 우려.
§1 현행 교직원 AI 업무 서비스

이미 상용 서비스로 공백을 메우고 있다 — 그것이 최소 기준선이다

① 최소 기준선

교직원은 이미 익숙하다 — 개통 시 이보다 못하면 그 순간 실패다.

② 검증된 요구사항

실사용 기능 구성은 업무 수요의 실증 자료 — ISMP의 SRS에 그대로 쓸 수 있다.

③ 브릿지

바당2 개통(2028-09)까지의 임시 수단 — 전환 관계(대체·병존·이관)는 확정 전 확인사항.

제주교육청은 2026년 교직원용 상용 AI 업무비서 「마이클」을 별도 도입해 운영 중이다(멀티 LLM 대화 · 한글 문서 AI 편집 · 업무 도구·챗봇 제작 · NEIS 학사·급식 연동 · 관리 콘솔). 교직원 전용이며 학생 기능은 없다.

바당은 처음부터 ‘끼우도록’ 설계된 개방형 플랫폼이다

바당2 — 모든 길은 관문을 지난다
게이트웨이 독점이 성립해야 기록·안전·통제가 성립한다 — 그 대가와 두 개의 금지선도 설계에 넣었다.
§3-1 설계 원칙 ① — 게이트웨이 독점

모든 AI 사용이 한 길목을 지난다

§3-1 설계 원칙 ② — 두 번째 금지선

추론은 벤더, 도구 실행은 코어

바당 샌드박스 — 실행은 우리 서버에서

  • 한 번 지어 함께 쓴다 — 「문서공방」(hwpx 생성) · 「샌드박스」(코드 실행)
  • 실행할 때마다 일회용 격리 상자가 뜬다 — 네트워크 차단 · 시간·메모리 상한 · 용도별 권한 프로파일
  • 벤더 런타임에는 hwpx 라이브러리가 없다

바당검색 — 검색도 코어를 지난다

  • 웹에 나가는 것은 우리 서버다 — 모델의 검색 요청을 코어가 받는다
  • 교육용 화이트리스트 · 연령 차등 필터를 걸어 걸러진 결과만 모델에게
  • 가져온 데이터도 코어 안전필터를 통과시킨다
벤더 SDK 하네스는 우리 서버에서 구동한다 — 에이전트 루프 코드를 밑바닥부터 짜지 않되, 매 턴이 코어를 지나 N개 로그·N번 비식별이 남는다.
§3-1 독점의 대가 — 두 리스크

한 길목에 모이면, 가용성과 품질이 값이다

① 단일 장애점(SPOF)

  • 코어 HA·무중단 배포·오토스케일
  • LLM 장애 → 타 벤더 폴백 · degrade 범위 사전 정의 · 서킷브레이커 → SLA 보장
  • 벤더별 기능 대응표 유지 · 분기 1회 폴백 전환 훈련
  • 게이트웨이 코어도 검증된 오픈소스 LLM 게이트웨이 기반 채택 우선 검토

② 우회 — 품질이 낮으면 독점이 무너진다

  • 품질이 낮으면 학생이 벤더 앱으로 샌다 → UI/UX 품질이 성패의 전제(기능 1)
  • 과제·평가는 바당2 경유만 인정(세션 토큰·산출물 표식)
  • MDM·네트워크 정책 — 기술만으론 100% 불가, 정책 + 기술 병행
“안전·로깅은 못 끄되, 추론은 멈출 수 있다”는 실패 모드를 인정하고 설계한다.
§3-2 바당 ↔ 바당2 연계 구조

표준 넷으로 결합하고, 저장 역할은 나눈다 (중복 구축 아님)

인증 — SAML 2.0SSO · 재로그인 없음
명부 — OneRoster학급 전체 명단 → 현황판이 ‘아직 시작 안 한 학생’까지 표시
도구·UX — LTI 1.3Deep Linking · 클래스 안에서 클릭 한 번에 SSO
기록·분석 — 바당2 완결대시보드·생기부 근거를 직접 제공 (xAPI 환류는 옵션)

저장 역할 분리

  • 학습기록 → 바당2 DB — 학습분석·대시보드·생기부 근거 (xAPI로 사상 가능하게 설계)
  • 송수신 전문 원본(raw 전량) → 교육청 로그 — 안전·감사·개인정보용. 바당엔 원래 없던 것 = 중복이 아니라 신설
  • 학습 상태 모델 · 실시간 운영 상태 → 바당2 DB

왜 나누는가

학습기록 저장소(LRS)는 기록을 쌓는 append 저장소이지, 매 턴 컨텍스트를 저지연으로 읽고 쓰는 실시간 대화 운영 DB가 아니다.

한 줄 정리 — 바당2는 단일 게이트웨이. 수업 화면만 바당의 「LTI 외부 도구」로 등록하고 업무 화면은 SAML 직접 접속.

§3-3 대원칙(불변)

새 모델·기능은 ‘코어 수정 없이’ 붙인다

  • 코어는 얇고 고정 — 인증·모델 라우터·안전필터·전량 로깅·에이전트 오케스트레이션·바당 연계. 길목은 독점하되 구체 구현은 코어에 박지 않는다.
  • 모델 = 프로바이더 어댑터 — 새 벤더·모델(향후 교육부 특화 LLM 포함)은 어댑터 추가 + 설정 등록으로.코어 라우터 무수정
  • 모달리티별 최적 엔진 조합 — 대화는 A사, 「이미지」은 B사, 「음성대화」는 자체 스택. 벤더 앱은 구조적으로 못 하는 일 — 각 벤더 앱은 자사 엔진만 쓴다.
  • 첫 번째 금지선 — 인증·라우팅·전량 로깅·비식별·안전필터·위기감지를 모델이 호출하는 도구(MCP)로 만들지 않는다. 도구는 모델이 호출 여부를 결정하므로, 안전 장치를 도구화하면 독점이 무너진다 — 이들은 건너뛸 수 없는 파이프라인 구간이다.
  • 코어도 늙는다 — 확장 마켓은 코어 기능의 ‘더 나은 대안’까지 허용하고, 무엇을 쓸지는 학교·교사가 고른다.코어가 낡아도 플랫폼은 늙지 않는다
공공은 토대, 민간은 속도 — 교육청은 예산·조달·심의 절차상 빠른 기능 변경이 구조적으로 어렵다. 공공은 안정적 토대(게이트웨이·표준 연계·로깅·안전·거버넌스)에 충실하고, 빠르게 바뀌는 응용 기능은 업체 확장에 맡긴다.

생태계는 표준 위에서 자란다. 문항을 QTI로 만들면 ① 에듀테크 업체는 가진 문항을 그대로 들고 마켓에 온다 ② AIDT·타 시도 콘텐츠를 그대로 수용한다 ③ 제주 콘텐츠를 타 시도·에듀테크로 내보낸다 — 제주의 콘텐츠가 표준 자산이 된다.

§3-5 거버넌스·개인정보

벤더에는 가명 토큰만 — 기록은 감시가 아니다

Mask가명 토큰으로 치환해 전송 — 기본값: 학생 세션
Mask + 재식별출력단에서 실명 복원 — 교사는 실명 문서를 받고, 벤더는 실명을 본 적이 없다. 기본값: 업무 세션
Block민감정보 포함 시 질문 차단 — 고위험 업무
해제(Off)전역 토글 없음 — 요청 단위 · CPO 승인 · TTL 만료 · 전량 감사

해제 불가 구간 — 어떤 승인으로도 열 수 없다

  • 만 14세 미만 학생의 정보 — 법정대리인 동의 + 확인(검증)을 이중으로 요구 · 민감정보(상담·정신건강·건강·위기신호)
  • 고유식별정보·주민번호·계좌·카드번호 · 프롬프트에 등장하는 제3자의 식별정보(이용자 전원 동의로도 원리적으로 메울 수 없다)
① 데이터 최소화가 1차 방어선 — 학생 식별정보는 애초에 프롬프트에 넣지 않는다(비식별 엔진은 2차). ② 동의는 안전조치 의무를 면제하지 못한다 — “동의받았으니 비식별 불필요”는 성립하지 않는다. ③ 기록 ≠ 상시 열람 — 교사 열람은 집계·요약 기본, 학습 중 오답·시행착오는 평가에 불리하게 쓰지 않는다. ④ 매핑 테이블 = 최우선 보안자산 — KMS/HSM · 2인 승인 · 전건 감사.
§3-5 국외이전 — 확인 완료된 사실

‘국내 리전으로 회피’하는 경로는 없다

확인된 사실 (2026-07)

  • AI 벤더가 국내 리전을 제공하지 않는다 → 국외이전은 확정 적용, 별도 동의가 사실상 유일한 근거
  • 혼동 금지 — 바당2 자체 인프라(웹·DB·API 코어)는 바당과 같은 CSAP 국내 리전. 국외로 나가는 것은 벤더로 보내는 가명 프롬프트뿐

그래서 두 갈래다

  • ⓐ 동의 체계를 운영 가능하게 설계 — 만 14세 미만(초등 전원 + 중1 일부)의 법정대리인 동의 + 확인(검증)을 가정통신문·NEIS 연계·학기별 갱신으로
  • ⓑ 연령별 모델 라우팅 — 국외이전이 없는 국내 벤더를 학생 세션 화이트리스트에. 어댑터라 코어 무수정
⚠️ 쓰지 말 표현 셋 — ❌ “마스킹은 법정 의무다” ❌ “동의를 받으면 적법하다” ❌ “개인정보위가 마스킹 해제를 금지했다”. 셋 다 근거가 없다. 방어 가능한 명제는 “동의는 안전조치를 면제하지 못하며, 마스킹은 그 채택된 수단”이다.
벤더 계약(DPA) 4개 필수 조항 — ① 학습 미사용 ② 제로 리텐션 ③ 국외 이전·재위탁 제한 ④ 침해 통지. 미충족 벤더는 어댑터 등록 불가.

기능 7 — AI 토큰 관리

cap의 목적은 절약이 아니다 — 예고 없이 끊지 않는다

cap 정책 — 「사전 요청 · 교사 수용 → 차단」

  • 80% 알림 → 95% 경고 + [추가 배정 요청] 버튼 → 요청 시 교사가 수용(수업 중 즉시) → 100% 차단
  • 교사가 수용 주체인 것은 교육적 판단이 필요하기 때문 — 관리자 결재로 올리면 수업 중 처리가 불가능하다

폭주(runaway)는 성격이 다르다

  • 정상 사용의 cap 초과 = 사용자가 원한 사용 → 경고 후 차단, 즉시 복구 경로
  • 비정상 폭주 = 아무도 원하지 않은 호출 → 경고 없이 즉시 중단
  • 무한 루프 · 재귀 팬아웃(깊이 1단계 제한) · 재시도 루프 · 확장 유발 폭주 · 반복 요청 남용
🟦 회계 구분 — API 사용료는 구축비가 아니다. ① 구축비(FP 산정 · 2027 본예산 · SW 개발)와 ② AI API 사용료(벤더 별도 계약 · 종량·선불 · 2028 본예산 시범 운영비)는 섞이지 않는다.

심사: 없음
