바당(AIEP)이란

표준 콘센트를 갖춘 ‘그릇’은 완성됐다

무엇인가

  • 11개 시·도 교육청 공동개발 LMS(NIA 위탁) — ’25.12.22. 오픈
  • 한 번의 로그인(SSO)으로 에듀테크·교육청 서비스 이용
  • 블렌디드 수업 지원 · 학습분석 대시보드

무엇을 갖췄나

  • 클래스·수업·기록 관리 — 콘텐츠 약 9만 개
  • NEIS 기반 학교 환경 자동 구성
  • SSO(SAML)·OneRoster·LTI·xAPI — 국제 표준 개방형 연계

공백 — 무엇이 비어 있나
학생들은 이미 생성형 AI를 쓰고 있다. 공교육의 장치는 아직 없다.
구축 현안 사항(’26.6.) — 공식 확인 사항

교육청 스스로 확인한 다섯 가지 한계

  • ① 연계 에듀테크 학습데이터 수집 불가 — 플랫폼 자체·빅테크 데이터 외에는 확보 못함.
  • ② 학습데이터 분석·활용 기능 미흡 — 2차 구축 시 고도화 필요.
  • ③ AI 모델 부재 — 실증만 완료. 플랫폼 내장 학생용 생성형 AI 없음.
  • ④ 에듀테크 사용권 반복 지출 — 매년 벤더별 계약비가 나가지만, 지출이 데이터·자산으로 남지 않음.
  • ⑤ 교육부 트랙과 중복·혼선 우려 — 유사 기능에 공적 재원 이중 투입 구조(지방비 24억 + 특별교부금 28억).
학생의 자리에서 — 공백 ⑥
0
학생이 수업 안에서 직접 쓸 대화형 AI — 없다.
연계된 교육용 생성형 AI(AI헬피챗)는 외부 벤더 서비스에 접속하는 형태. 주요 벤더의 소비자용 앱은 약관상 연령 제한이 있어, 미성년(특히 초등)에게 학교가 공적으로 제공하기 어렵다.
학생들은 이미 쓰고 있다생성형 AI 사용은 일상 — 지금은 공적 보호·기록 없이 각자 쓰는 중.
필요한 것 — 공적 대화형 AI모든 학생이 가정 형편과 무관하게, 교육청이 학교급·연령별로 정한 모델·기능 정책 안에서.
교사의 자리에서 — 공백 ⑦

교사가 AI를 수업 도구로 통제할 수단이 없다

지금 — 통제 수단 없음

  • AI의 응답 방식을 수업 목적에 맞게 제어할 수 없음 — 정답 제시형 그대로
  • 실시간 수업에서 AI를 활동 도구로 쓸 수단 없음

필요 — 수업 단위 설계·통제

  • 응답 방식 제어 — 튜터·소크라테스식 등
  • RAG — 교육과정·교재를 근거로 답하기
  • AI 에이전트 · MCP — 학습 이력 조회, 다단계 과제 수행 등 도구 연동
교사 설정은 학생이 우회할 수 없도록 게이트웨이(서버) 단계에서 강제 적용되어야 실효가 있다.
교육청의 자리에서 — 공백 ④⑧⑨

돈은 매년 나가는데, 데이터는 남지 않는다

④ 반복 지출 · ⑧ 송수신 전문 확보 불가 · ⑨ AI 학습활동이 기록(LRS·누가기록)으로 이어지지 않음
필요한 방향 — 모든 AI 사용이 교육청 관문을 경유해 기록·관리되는 데이터 주권 구조 (열람은 목적 제한 · 역할별 차등으로 학생 프라이버시와 양립).
보호의 자리에서 — 공백 ⑩⑪

미성년 보호 책무를 이행할 장치가 없다

안전장치 ⑩

  • 교육청이 기준을 정해 설정·검증·통제하는 연령별 필터·위기감지가 없음
  • 벤더 앱을 직접 쓰는 동안에는 공적 보호·기록이 미치지 못함
  • 필요 — 위기 신호는 확인 절차를 거쳐 담임·전문상담(Wee) 연계, 필터 우회·프롬프트 인젝션은 양방향 검사로

개인정보 ⑪

  • 외부 전송이 학운위 심의·동의 등 행정 절차에 의존 — 기술적 통제 수단 없음
  • 필요 — 전송 전 PII 마스킹·가명처리(자동 탐지의 한계 전제, 최소 전송 병행)
  • 벤더와 학습 미활용·보존기간 제한 계약 — 기술·계약 이중 통제, 원본은 국내·교육청 보관
공백 맵

아홉 가지 공백 — 실은 하나의 문제다

⑥ 학생용 생성형 AI수업 안에서 쓸 공적 대화형 AI 부재
⑦ AI 수업 도구교사의 수업 단위 설계·통제 수단 부재
⑧ 송수신 데이터AI 대화 전문의 확보·관리 불가
⑨ 기록 연계AI 학습활동 → LRS·누가기록 미연결
⑩ 안전장치연령 필터·위기감지·알림 부재
⑪ 개인정보전송의 기술적 통제(마스킹) 부재
⑫ 비용·한도사용량(토큰) 관리·모델 라우팅 부재
⑬ 접근성TTS·STT·다국어·저사양 지원 미흡
⑭ 품질 관리오답·환각 점검 체계 부재
아홉 가지는 서로 다른 문제가 아니다 — ‘AI 접근을 공적으로 다루는 층위’가 없다는 하나의 문제다.

방향 — 어떻게 채우나
공동 트랙에 요구할 것과, 제주가 직접 채울 것을 나눈다.
대응의 두 갈래

공동 트랙에 요구하고, 제주가 직접 채운다

공동 플랫폼 트랙 — 2차 구축에 요구

  • ① 데이터 수집 확대 — 에듀테크 xAPI 회송 구현
  • ② 학습분석 고도화 · ③ AI 모델 개발
  • 교육부 체계와 역할 분담 — 중복 투자·혼선(⑤) 예방

제주가 직접 — 표준 위에 얹는 계층

  • 학생용 생성형 AI 접근 · 데이터 주권 · 안전장치 · 비용·품질·접근성(⑥~⑭)
  • 공동개발 합의 구조·2차 일정으로는 채우기 어려운 과제들
특히 ⑧(데이터의 교육청 기록·관리)·⑩(직접 운영 안전장치)·⑫(통합 비용 관리)는 구조상 개별 교육청 단위로 계획되기 어려운 성격 — 확인된 양 트랙 계획 문서에도 없다.
결합 방식

재구축이 아니라, 표준 콘센트에 플러그를 꽂는다

사용자·클래스 맥락은 표준(LTI)으로 전달 — 학생은 바당 계정 그대로, 수업 화면 안에서 클릭 한 번
코어 수정 없이새 모델·기능은 어댑터 추가로 — AI 변화에 기민하게 대응.
중복 구축 없이명부·계정·LRS는 바당 것을 그대로 — 예산·혼선을 줄인다.
맺음
그릇은 완성됐다.
이제 알맹이를 채울 차례다.
바당이 이미 갖춘 표준 연계 위에, 학생용 생성형 AI 접근·데이터 주권·미성년 보호 안전장치를 제주가 직접 얹는다 — 이것이 제주교육 AI 플랫폼 구상의 출발점이다.
모든 학생의 AI 접근을, 공교육이 안전하게 보장한다.
