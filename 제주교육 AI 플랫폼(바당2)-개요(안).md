# 제주교육AI플랫폼(바당2) 개요

> *정식 명칭: **「제주교육AI플랫폼」** · 시스템/브랜드명: **바당2**(영문 표기 **BADANG2**, 이하 본문은 "바당2").*

> ### 🟦 한 줄 요약
> **「제주교육AI플랫폼」(바당2) = 로그인 한 번으로 *AI 대화·실시간 채팅 수업은 물론 교직원 업무 처리까지* 하고, *그 모든 활동을 DB에 기록*해 *분석·맞춤 데이터·콘텐츠로 되돌려주는* 제주교육공동체(학생·교사·교직원) 전용 공공 「AI 활용 플랫폼」.**
> *접근 → 활동 → **전량 기록** → 분석 → **맞춤 환류**. 단순 AI 챗봇이 아니라, 데이터가 쌓여 수업·업무로 되돌아오는 **선순환**이 핵심. (기존 LMS '바당'이 **그릇**이라면, 바당2는 거기 채우는 **AI 알맹이 + 안전장치**.)*
> *목적은 둘이며 **동급**이다 — ① **교육 특화**(초개별화·완전학습 루프) ② **범용 AI 활용의 공적 보장**(교육공동체 전원이 벤더 유료 구독 없이 최고급 AI를 쓴다). ②가 없으면 ①도 서지 않는다.*

> *작성 2026.6.13. · 개정 2026.7.12. · 미래학력분과. 기획용 **추진 계획(안)**이며, 사업범위·일정은 재원 확보(T0)·RFP·AIEP 2차 구축 일정과 정합을 맞춰 확정.*
> *연계 문서: **[AI 교육 패러다임](AI교육-패러다임.md)**(왜·무엇을 — 본 문서는 그 실현·구현) · **[바당 현황 및 개선 방안](<인공지능 맞춤형 교수학습 플랫폼 바당 현황 및 개선 방안.md>)**(현황 진단·공백 ①~⑭) · 「바당에서 제주교육AI플랫폼(바당2)으로」(내부 문서) · **「운영 계획으로 본 바당 격차·타당성」(내부 문서)**(교육청 공식 문서 인용만으로 세운 추진 근거) · 연계 구성도는 §2.*

## 0. 개발 배경

> **왜 지금 이 플랫폼인가 —** 공교육은 시대와 환경에 맞는 교육 인프라를 직접 조성해, 모든 학생이 그 시대의 핵심 역량을 **차별 없이** 갖추도록 책임져야 한다. 지금 그 핵심 역량은 **AI 활용**이며(학생은 이미 AI를 쓰고 있다), 이를 **안전하고 형평하게** 보장하려면 실시간 수업·AI 접근·안전·기록을 한데 갖춘 **공공 수업 플랫폼**이 필요하다 — 바당2가 그 인프라다. *(여기 압축한 '왜·무엇을'의 전체 논의는 [AI 교육 패러다임](AI교육-패러다임.md) 참조.)*

**바당과 바당2의 관계**

- **바당(BADANG) = AIEP**(인공지능 맞춤형 교수학습 플랫폼): 11개 시·도 교육청 공동개발(NIA 위탁), **2025.12.22 오픈한 기존 LMS** — 클래스·수업·배정, 학습분석, 에듀테크 SSO 연계를 갖춘 **그릇**.
- **바당2 = 바당 위에 얹는 「AI 접근·추론 게이트웨이 계층」** — **모든 학생의 AI 접근을 공적으로 보장**한다는 본 사업의 원칙을 실현하며, 그릇에 비어 있는 **AI 알맹이(생성형 추론)와 그 데이터·안전 장치**를 채운다.

| 구분 | 바당 (AIEP) | 바당2 (AI 게이트웨이 계층) |
| --- | --- | --- |
| 정의 | **LMS** — 학습관리 플랫폼(그릇) | 바당 위에 얹는 **AI 접근·추론 게이트웨이**(알맹이) |
| **소유·대상** | 11개 시·도 **공동**(NIA 위탁) | **제주교육공동체 전용** — 제주 직접 소유·통제 |
| **변화 대응** | 공동개발 → 변경에 다자 합의(경직·지연) | **AI 변화에 기능 확장·변경 용이**(제주가 기민하게) |
| 핵심 역할 | 명부·계정·권한 / 클래스·수업·배정 / 표준 연동 / 학습기록·분석(xAPI·LRS) | AI 접근 공적 보장, **송수신 전체 로깅**, 안전필터, AI 수업도구 |
| AI | **없음**(2차 구축 과제, 현안 명시) | **있음** — 본 계층이 제공 |
| 데이터 | 학습기록 **정본 저장소(LRS)** 보유 | AI 송수신 **전문 원본 확보**(신규 교육청 로그) · **학습기록·분석은 바당2 대시보드에서 완결**(xAPI 환류는 전면시행 옵션) · 세션·운영 상태 자체 보유 |
| 안전·거버넌스 | AI 통제 장치 없음 | 안전필터·위기감지·개인정보 비식별·접근권한 |
| 관계 | 표준 연계 그릇(SAML·OneRoster·LTI·xAPI) | **SSO·명부·LTI 표준으로 바당에 결합** — **LMS 재구축 아님** |

> **이용 대상·활용 범위:** **제주교육공동체 전체**의 AI 활용 플랫폼 — **학생·교사는 수업**(AI 대화·실시간 채팅 수업·기록·수업설계·평가, §4), **교직원은 행정 업무**(공문 초안·요약·분석)를 **동일 게이트웨이**(전량 로깅·안전·PII)로 쓰되, 권한·안전·보안정책은 분리한다(§6). 누구나 **기본 기능**(범용 AI 대화, 기능 1)으로 바로 쓰고, 특화 기능은 **확장팩**(마켓, 기능 10)으로 골라 장착한다 — §3-3 *코어=기본 보장선/확장=특화·대안*. **개통일부터 마켓은 비어 있지 않다** — 교육청이 제작한 **「개통 기본 스킬 세트」**(학생 학습 · 교사 수업·평가 · 교직원 업무, 기능 10)가 처음부터 진열된다.

> **목적 ② — 범용 AI 활용의 공적 보장 (교육 특화와 동급):** 바당2는 교육 특화 기능만의 플랫폼이 아니다. **제주교육공동체 전원에게 최고급 범용 AI 활용 환경을 공적으로 보장**하는 것이 동급의 목적이다. *왜 벤더 구독을 사주지 않고 직접 만드는가* 에 대한 답이 여기 있다 — **구독으로는 기록도·안전도·형평도·공문서(hwpx)도 얻을 수 없고, 그 비용은 매년 반복**되기 때문이다(§1의 다섯 공백이 그대로 재현된다).
>
> **정부 정책과 정면으로 일치한다 — 「AI가 읽는 행정」.** 「행정업무 운영 및 혁신에 관한 규정」 개정(2026.5.12. 국무회의 의결)으로 **2026.10.부터 공공기관 기본 문서 포맷이 hwpx로 의무화**된다. **그 전환 이유가 곧 "AI가 읽을 수 있는 문서를 만들기 위해서"** 다 — **정부가 AI가 읽을 문서를 준비했고, 바당2가 그 AI다.** 개방형 포맷 위에서 공문·규정·수업자료를 읽고 쓰는 공공 AI를 교육청이 직접 갖는 것이 이 사업이다(기능 1 「문서공방」·「한글 편집기」 · 기능 10 「규정·지침 질의」).
>
> **이 목적의 검수 지표 — 「벤더 구독 재가입 최소화」:** *개통 시점 기준, 교직원·교사가 일상 업무와 수업 준비를 수행하는 데 **벤더 유료 구독을 별도 가입할 필요가 없어야 한다.*** 시범 운영 중 **"바당2로 안 되어 벤더 앱을 따로 썼다"는 사례를 수집·계량**하고, 그 목록이 곧 다음 릴리스의 백로그가 된다. 이는 §3-1의 '우회 차단'(위험 관리)과 층위가 다른 **제품 목표**이며, 기능 1의 「개통 필수 목록」이 그 실현 수단이다.

**국내외 위치 — 새 발명이 아니라 '첫 온전한 조립'**

> 바당2의 식별 특징은 *모든 AI 사용이 통과하는 공공 게이트웨이*이며, 다음 **8요소의 결합**이다: ① 공공 소유 게이트웨이 · ② **벤더앱급 UX·기능 집합**(기능 1 「개통 필수 목록」이 그 실체 — 도구 실행은 코어에서, §3-1) · ③ 전량 송수신 로깅 · ④ PII 비식별(**텍스트 + 비전**, §3-5) · ⑤ 다중 LLM 라우팅·폴백(**모달리티별 최적 엔진 조합**, §3-3) · ⑥ 안전필터·위기감지 · ⑦ 토큰·비용 통제 · ⑧ 교사·확장 생태계. 국내외 20여 개 플랫폼을 같은 자로 재본 결과, **이 8요소를 모두 갖춘 사례는 공개 자료상 확인되지 않는다** — **세계가 조각으로만 운영해 온 패턴을 교육 맥락에서 처음으로 온전히 조립**하는 사업이다.
> 가장 근접한 사례는 🇸🇬 싱가포르 SLS(모델무관 가드레일 게이트웨이 — 전량 로깅·비식별은 탐지 수준), 반면교사는 🇺🇸 美 LAUSD "Ed"(벤더 소유·비식별 부재 — §3-1). 상세 비교·출처는 **「유사 플랫폼 리뷰」(내부 문서)** 참조.

> **현행 교직원 AI 업무 서비스 — 공백을 임시로 메우고 있는 상용 서비스:** 위 공백 때문에, 제주교육청은 **2026년 교직원용 상용 AI 업무 서비스를 별도 도입해 운영 중**이다(멀티 LLM 대화 · 한글 문서 AI 편집 · 업무 도구·챗봇 제작 · NEIS 학사·급식 연동 · 사용량·정책 관리 콘솔). **철저히 교직원 전용이며 학생 기능은 없다.**
> - **이 사실은 바당2 설계에 두 가지를 의미한다.** ① **바당2 「업무 세션」의 최소 기준선** — 교직원은 이미 이 서비스에 익숙하므로, **개통 시 이보다 못하면 그 순간 실패**한다. ② **현장 검증된 요구사항 명세** — 실사용 중인 기능 구성은 교직원 업무 수요의 실증 자료이며 ISMP의 SRS에 그대로 쓸 수 있다.
> - **전환 관계:** 현행 서비스는 **바당2 개통(2028-09)까지의 브릿지**다. 개통 이후의 관계(대체·병존·단계적 이관)와 그때까지의 사용 데이터 활용은 §6에서 다룬다.
> - *(상품명·업체명은 발주 공정성을 위해 본 문서를 포함한 모든 정식 문서에 기재하지 않는다. 기능 전수와 격차 도출은 **[현행 교직원 AI 업무 서비스 기능 분석](<참고자료/현행-교직원AI업무서비스-기능분석.md>)** 참조.)*

> **요약:** 바당은 **LMS(그릇)** 는 갖췄지만, **① 학생용 생성형 AI ② 송수신 데이터 확보 ③ 안전·거버넌스**가 비어 있다. 바당2는 정확히 이 세 공백을 메운다. 「현황 및 개선 방안」은 이를 아홉 공백(⑥~⑭)으로 세분화했으며, 공백↔기능 대응은 「바당에서 제주교육AI플랫폼(바당2)으로」(내부 문서)의 대응표 참조.

## 2. 바당(AIEP) 시스템 구성도 — 표준 기반 개방형 연계

> 근거: 「구축 현안 사항」 [붙임1] **인공지능 맞춤형 교수학습 플랫폼 시스템 구성도**.

![바당 · 바당2 연계 구성도](<AI교육-바당2-연계구성도.svg>)

*[그림] 바당(AIEP·LMS) ↔ 바당2(AI 게이트웨이) 표준 연계 구성도 — 추론은 외부 벤더, 인증·명부·도구 실행은 바당 표준(SAML·OneRoster·LTI)으로 결합. 기록·분석은 바당2에서 완결(점선 = 전면시행 옵션인 xAPI 환류, §3-2).*

바당은 처음부터 **외부 시스템을 표준 API로 끼우도록(plug-in)** 설계된 개방형 플랫폼이며, 구성도에 연계 표준이 명시되어 있다.

| 연동 영역 | 표준 / 구성요소 |
| --- | --- |
| 인증(SSO) | **SAML 2.0** · 통합인증 / 간편 로그인 |
| 학생정보(명부) | **OneRoster V1.1** (통합형 학생정보 시스템) |
| 도구·서비스 연동 | **LTI / API** (외부기관·에듀테크·콘텐츠 연동) |
| API 계층 | **API Gateway · API 포털 · OpenAPI · API 모니터링** (연계시스템 영역) |
| 학습기록 | **xAPI · LRS(학습기록저장소)** · LOM 메타데이터 |
| 외부연계 대상 | **AIDT(AI디지털교과서) · 11개 교육청 표준기반 기존시스템 · 외부 LMS · 에듀테크/콘텐츠 Provider** |
| 채널 | 네이버 웨일 · MS Teams · Google Classroom · Apple ASM · 민간 LMS |
| 인프라 | 민간형·멀티 클라우드 네이티브 |

> **시사점:** 바당은 AIDT·에듀테크·외부 LMS를 **이미 표준 API로 연동**하고 있다. 따라서 **바당2도 "또 하나의 외부 도구"로 이 표준 연계 계층에 붙으면 된다** — 바당 소스코드를 복제(fork)·수정하는 것이 아니라 밖에서 끼우는 결합이므로 NIA·타 시·도와의 코드 충돌·라이선스 문제가 없고, 새 플랫폼을 따로 세우지 않으니 사업 중복·혼선 우려(§1-⑤)도 줄어든다. *(구체 결합 방식·사용 범위는 §3-2.)*

## 3. 바당2 개발 개요

### 3-1. 정의와 위치

- 기존 바당(LMS) **위**에 얹는 **AI 접근 게이트웨이 계층**. 학생·교사의 **모든 AI 사용이 이 게이트웨이를 통과**한다.
- **설계 원칙(게이트웨이 독점, B안):** 모든 길목을 게이트웨이가 독점해야 **전(全) 송수신 로깅·안전필터·통제**가 가능. 벤더 앱 직접 사용(A안)으로는 송수신 전문 확보·통제 불가 → 본 사업의 직접 근거.

> **반면교사 — 검증된 국내외 선례:**
> - **🇺🇸 LAUSD "Ed"(AllHere) — 최우선 반면교사.** LA통합교육구는 벤더 **AllHere Education**과 **5년 약 600만 달러** 계약으로 2024.3. AI 비서 "Ed"를 출시했다. **AllHere의 前 소프트웨어 엔지니어링 시니어 디렉터(내부고발자)** 는 *"**데이터를 해외로 보내고, 너무 많은 데이터를 보내며, 제3자가 그 데이터를 로깅한다**"* 며 **교육구의 데이터 이용 협약 위반**을 고발했고, 그가 공개한 **소스코드**상 프롬프트는 **해외 서버**(ChatGPT 연동 상용 AI 서비스)에서 처리되고 있었다. 그는 교육구·감사관실·주 교육당국에 알렸으나 **언론 보도 전까지 응답을 받지 못했다.** AllHere는 직원 대부분 무급휴직 후 **파산(Chapter 7)** 했고 Ed는 중단됐으며, **2024.9. 미 법무부가 대배심 소환장을 발부해 연방 형사 수사**로 이어졌다.
> - **교훈 셋:** ① **벤더 소유 인프라는 벤더와 함께 무너진다** — 공공이 게이트웨이·로그·데이터를 소유해야 한다 ② **「데이터 이용 협약」은 계약서일 뿐 강제 장치가 아니다** — **비식별이 기술로 강제**되지 않으면 협약은 지켜지지 않는다 ③ **내부 경고가 무시됐다** — 검수·운영에 **작동하는 경고 채널**이 있어야 한다. 위험은 게이트웨이 패턴 자체가 아니라 **누가 소유하며 비식별이 실제로 강제되는가**다. 바당2의 **공공 소유 · 게이트웨이 독점 · 강제 비식별(§3-5) · 국내 보관**은 이 실패를 구조적으로 차단하는 설계다.
> - **🇰🇷 AI 디지털교과서 사전 실태점검(2025.5) — 교육 도메인 직접 선례.** 개인정보보호위원회는 학생별 학습이력(학습시간·성취수준·진도율·접속시간·커뮤니티 참여도)을 통합 DB에 누적하는 구조에 대해 **"학생의 일상행동 감시 등 오남용 우려"**·처리목적 미정립을 지적하고 **법 §3①(목적 명확화·최소처리 원칙) 위반으로 시정권고**했다(KERIS 대상 · 교육부 개선권고 병행). **교훈: 「교육 목적」이라는 선언만으로는 학생 데이터의 광범위한 축적이 정당화되지 않는다** — 목적을 항목 단위로 명확히 하고 최소처리를 증명해야 한다. 바당2의 **데이터 최소화(§3-5)·목적 제한(기능 3 거버넌스)** 이 이에 대응한다.
> - **🇰🇷 이루다 사건(2021.4. · 개인정보보호위원회 과징금·과태료 부과).** 포괄·약관형 동의는 특정 목적의 적법 근거로 인정되지 않았고, **학습데이터 내 이름·전화번호·주소를 삭제·암호화하지 않은 것 자체가 유책 사유**였다. **교훈: 「동의를 받았다」는 형식적 방어는 깨진다 — 안전조치(비식별)는 동의와 별개의 병존 의무다**(§3-5).
> - **🇮🇹 Replika 제재(Garante, 2025.4. — 500만 유로).** 처리행위별 **적법 근거 미특정** · 부정확한 처리방침 · **미성년자 연령확인 부재**가 핵심 위반이었다(GDPR — 국내 규제기관을 구속하지 않는 **설득적 권위**). **교훈: 미성년 대상 AI 서비스에서 「연령 확인」과 「근거의 특정」은 형식이 아니라 제재 사유다**(기능 5 · §22-2 대응).
>
> **공통 교훈:** 위험은 게이트웨이 패턴 자체가 아니라 **① 누가 소유하며 ② 비식별이 실제로 강제되고 ③ 처리 근거가 항목별로 특정되는가**다. 바당2의 **공공 소유 · 게이트웨이 독점 · 강제 비식별(§3-5) · 국내 보관**은 이 셋에 대응하는 설계다.
>
> ※ **선례 인용 원칙:** **국내 발주 문서에서는 국내 규제기관의 선례가 해외 사례보다 강한 권위를 갖는다.** 해외 사례(Replika 등 GDPR)는 **설득적 권위로만** 쓰고 조문 근거는 개인정보보호법으로 건다. 상세는 **[외부 LLM 전송의 개인정보보호법 쟁점 — 조사 결과](<참고자료/외부LLM전송-개인정보보호법-조사결과.md>)**.
- **AI 추론은 외부 API에서 수행:** 외부 LLM API(예: Claude 등)로 추론 → **교육청 서버에 LLM 추론용 GPU 팜 불필요.** 모델 라우팅으로 경량·표준·고성능 모델 혼합. **모델은 어댑터 방식으로 연동 — 새 모델은 코어 수정 없이 추가**(§3-3 대원칙).
 - **"GPU 불필요"는 LLM 추론용 GPU에 한정한 진술이다.** 아래 「추론은 벤더, 도구 실행은 코어」 원칙과 §3-5의 **비전 비식별**(얼굴·이름표 탐지)을 도입하면 **경량 비전 가속기(소수)** 는 필요하다 — LLM 추론용 대형 GPU 팜과는 규모·성격이 다르다(§3-4 부하 축 ④).

- **설계 원칙(두 번째 금지선) — 「추론은 벤더, 도구 실행은 코어」:** 벤더가 제공하는 **서버 실행형 도구**(코드 실행 · 웹 검색 · 파일 생성 등)를 **사용하지 않는다.** 그 도구들은 **벤더 인프라에서 실행**되므로, 학생이 올린 파일·코드·검색어가 벤더 서버로 넘어가 **전량 로깅·비식별·국내 보관이 그 지점에서 끊긴다.** 동일 기능은 **코어 내 자체 구현**(격리 샌드박스·검색 프록시)으로 제공한다 — 기능 1 「개통 필수 목록」이 그 구현 위치를 명시한다.
 - **이는 우회로가 아니라 정공법이다.** 주요 벤더 API는 bash·텍스트 에디터·컴퓨터·메모리 도구를 애초에 **호출자가 실행하는 방식(client-side tool)** 으로 정의한다 — 벤더가 스키마만 주고 실행은 넘긴다. **벤더 서버 실행형(code execution·web search)이 오히려 편의 옵션**일 뿐이다.
 - *(§3-3의 첫 번째 금지선 — "안전 장치를 모델이 호출하는 도구로 만들지 않는다" — 과 축이 다른 별개 원칙이다. 첫째는 **모델이 안전을 건너뛰지 못하게**, 둘째는 **데이터가 코어를 벗어나지 못하게** 한다.)*
- **독점의 대가 ① 가용성(SPOF 방어):** 모든 AI 사용이 한 길목을 지나므로 게이트웨이는 **단일 장애점·병목**이다 → **코어 HA·무중단 배포·오토스케일**, **외부 LLM 장애 시 폴백**(타 벤더 라우팅)·**degrade**(읽기전용 안내)·서킷브레이커·타임아웃·재시도를 **SLA로 보장**(RFP 필수). "안전·로깅은 못 끄되 추론은 멈출 수 있다"는 실패 모드를 인정하고 설계한다. 폴백은 선언이 아니라 운영이다 — **벤더별 기능 대응표**(멀티모달·도구호출·구조화 출력)를 유지하고 폴백 시 기능 축소(degrade) 범위를 사전 정의하며, **분기 1회 이상 폴백 전환 훈련**을 SLA에 포함한다 — 대응표 유지·훈련 수행 주체는 **지속 개선 계약의 과업 범위에 명시**한다(운영 이관 후 방치 방지). 게이트웨이 코어도 기능 1과 같은 원칙으로 **검증된 오픈소스 LLM 게이트웨이 기반 채택을 우선 검토**한다(라우팅·폴백·비용 집계는 이미 상품화된 문제).
- **독점의 대가 ② 우회 차단:** 품질이 낮으면 학생이 개인기기·무료 벤더앱으로 **우회**해 로깅 독점이 무너진다 → 교실 **과제·평가는 바당2 경유만 인정**(세션 토큰·산출물 표식) + **MDM·네트워크 정책 연계**. 기술만으론 100% 불가하므로 **정책+기술 병행**.

### 3-2. 바당 ↔ 바당2 연계 구조 (표준 기반)

바당2는 §2의 바당 개방형 **표준 연계 계층**에 결합한다. 추론만 외부 LLM이 수행하고, **인증·명부·도구 실행은 바당의 표준으로 물리되, 학습기록·분석은 바당2 안에서 완결**한다(독립 운영 — 바당 개선·2차 구축 일정과 얽히지 않는다).

**연계 사용 범위 요약 (시범 단계 확정)**

| 연계 | 사용 | 역할 |
| --- | --- | --- |
| SAML SSO | ✅ 사용 | 바당 로그인 공유 — **계정·비밀번호 인프라 불요** |
| OneRoster | ✅ 사용 | 학급 명부 자동 수신 — **교사 학급 개설 수작업 불요**, 현황판 미시작 학생 표시 |
| LTI 1.3 | ✅ 사용 | 바당 클래스 안 **한 클릭 실행**·차시 배치 |
| API Gateway | ✅ 사용 | 표준 호출 통로 |
| **xAPI 환류(바당 LRS 적재)** | ❌ **시범 미사용** | **학습기록·분석·현황판·생기부 근거는 바당2 대시보드에서 완결** — 전면시행 단계 옵션(AIEP 2차 구축 결과·현장 수요 확인 후 결정) |

> **이 범위 설정의 효과:** ① 바당에서 **받아오는 연계**(인증·명부)와 도구 등록(LTI)만 남아 NIA 협의에서 가장 무거운 항목(바당 LRS **쓰기** 권한)이 빠지고 — 남는 쓰기 후보는 **성적 회송(LTI AGS)** 하나뿐인데 이는 LTI 도구 등록에 딸린 표준 채널이라 LRS 쓰기와 무게가 다르다(지원 여부·미지원 분기는 §6) — ② 바당 2차 구축('26.8~'27.2)의 규격 변동 영향이 최소화되며 ③ **교사가 보는 화면이 바당2 대시보드 하나로 명확**해진다. 연결은 표준 그대로, **연결 이후의 처리(안전필터·로깅·보관·열람·모델 선택)는 전부 교육청이 결정**하는 구조다.

| 인증 | **SAML 2.0 SSO** | 바당 로그인 공유 — **클래스 밖 직접 접속**(교직원 업무·독립 사용)은 SAML SSO, **클래스 안 진입은 LTI 1.3 실행이 곧 인증**. 두 경로의 사용자는 **동일 신원(OneRoster 식별자)으로 매핑**해 기록·상태를 하나로 누적 |
| 명부 | **OneRoster V1.1** | 클래스·수업·학생 배정 정보 **사전 수신** — 접속한 학생만이 아니라 **학급 전체 명단**을 확보(기능 3 현황판이 '아직 시작 안 한 학생'까지 표시하는 근거 · 개별 실행 맥락은 LTI가 전달) |
| 도구 연동·UX | **LTI 1.3 (LTI Advantage)** | 바당2를 **LTI 1.3 도구로 등록** → 학생이 바당 클래스 안에서 **클릭 한 번에 SSO된 채** 사용 · 교사는 **Deep Linking**으로 바당 수업 차시에 바당2 활동(튜터 모드 등)을 배치 |
| 기록·분석 | **(바당2 자체 완결)** | 학습기록·분석·현황판·생기부 근거는 **바당2 대시보드·DB에서 제공**(xAPI 환류는 위 표). 생기부는 교사 검토·승인 참고자료(자동 기재 아님, §5 거버넌스 확정 사항) |
| 호출·관리 | **API Gateway / API** | 바당 연계시스템(API Gateway) 경유로 표준·안전 호출 |

> **한 줄 정리:** 바당2 = 단일 게이트웨이 — **수업 화면만** 바당의 「LTI 외부 도구」로 등록하고 업무 화면은 SAML 직접 접속(§3-6) · 기록·분석은 바당2에서 완결.

> **저장 역할 분리 (중복 구축 아님):** 기록을 두 종류로 나눈다.
> - **학습기록:** 정본은 **바당2 DB** — 학습분석·대시보드·생기부 근거를 바당2가 직접 제공(표준 LRS 제품을 재구축하는 것이 아니라 운영 DB·대시보드 형태). 학습활동 이벤트 스키마는 처음부터 **xAPI로 사상(mapping) 가능하게 설계**하고 어휘(동사·활동 타입)를 xAPI 프로파일로 정의해 둔다(RFP 요구사항) — 환류(위 표)를 켜는 순간 변환기만 추가하면 되도록. *(저장 포맷 자체를 xAPI statement로 하라는 뜻이 아니다 — 조회·집계는 운영 DB가 맡고, xAPI는 내보내기 계층.)*
> - **송수신 전문 원본(raw 전량):** 안전·감사·개인정보용. 바당엔 **원래 없던 것**(§1-②)이라 바당2가 **신규 확보** → **교육청 소유·국내 보관**(§3-5). LMS 중복이 아니라 **신설**이다.
> - **학습 상태 모델(지식 추적, 기능 3):** 학생별 숙달 상태를 성취기준·스킬 단위로 **갱신·보존하는 파생 모델** — 바당2 DB 보유. 학습기록의 *요약·현재값*이지 기록 자체가 아니며, 갱신 근거가 된 평가·도달 이벤트는 학습기록(바당2 DB)에 적재하고 **상태 모델은 그 기록으로부터 재산출 가능**하게 설계한다(정본은 어디까지나 증거 = 학습기록·전문 로그).
> - 그 밖에 바당2가 쥐는 것은 **실시간 운영 상태**(세션·컨텍스트 · 안전 이벤트·알림 · 식별↔익명 매핑 · 토큰·비용·레이트리밋) — §3-4의 PostgreSQL·Redis.
>
> **이유:** 학습기록 저장소는 기록을 *쌓는* append 저장소이지, **매 턴 컨텍스트를 저지연으로 읽고 쓰는 실시간 대화 운영 DB가 아니다** → **학습기록은 비동기 적재, 실시간 대화는 바당2 로컬 세션**으로 처리하는 것이 정석(전면시행에서 xAPI 환류를 켜는 경우에도 동일 원칙).

### 3-3. 모듈형·확장 아키텍처 (설계 대원칙)

> **대원칙(불변): 새 모델·기능은 "코어 수정 없이" 추가한다.** 모든 것을 인터페이스 뒤에 끼우는 **어댑터 방식**으로 설계한다. 이 원칙이 흔들리면 모델·기능이 늘 때마다 메인 프로그램을 고쳐야 해 유지보수·확장이 무너진다. *(이는 "교사가 켜고 끄느냐"와 무관한, 구현 방식의 문제다.)*

- **코어(게이트웨이 · 얇고 고정):** 인증·SSO · **모델 라우터** · 안전 필터 · **전체 송수신 로깅** · 데이터 적재 · **에이전트 오케스트레이션**(아래) · 바당 연계 — 모든 AI 사용의 길목을 독점하되, **구체 구현(어떤 모델·어떤 기능)은 코어에 박지 않는다.**

- **에이전트 오케스트레이션 — 코어 구성요소다:** §4 서두는 이미 *"**하네스 = 코어 게이트웨이** — 이 에이전트들을 안전하게 구동·기억·기록·환류하는 공통 실행 틀"* 이라고 선언했으나, **정작 코어 구성 목록에 이 층이 없었다.** 여기서 정식화한다.
 - **벤더가 파는 것은 「한 번의 대화」뿐이다.** 여러 대화를 **편성·병렬 실행·교차검증·합성**하는 것은 **평범한 서버 코드**이며 특별한 AI 기능이 아니다 — **전적으로 우리가 만든다.**

| 층 | 무엇 | 누가 |
| --- | --- | --- |
| **추론** | 에이전트 한 명의 사고 | **벤더** (LLM API 호출 1회) |
| **도구 실행** | 그 에이전트의 검색·코드실행·파일생성 | **코어** (§3-1 두 번째 금지선) |
| **오케스트레이션** | **몇 명을·어떤 역할로·병렬인지·누가 누구를 검증하는지·결과를 어떻게 합칠지** | **코어 — 전적으로 우리** |

 - **⚠️ 벤더의 서버형 오케스트레이션 상품은 사용 금지.** 벤더가 에이전트 루프를 대신 돌리고 도구 실행 컨테이너까지 호스팅하는 상품이 존재하나(편의 옵션), 그것을 쓰면 **N번의 송수신·도구 실행·중간 산출물이 전부 벤더 인프라 안에서 일어나고 코어는 최종 결과만 본다** — 전량 로깅·비식별·국내 보관이 그 지점에서 끊긴다. **호출이 N배이므로 §3-1 위반의 대가도 N배다.** 자체 오케스트레이션이면 **N번의 호출이 모두 코어를 지나 N개의 로그가 남고 N번의 비식별이 걸린다.**
 - **구성요소 여섯 (RFP 필수 요건):**

| 구성 | 하는 일 |
| --- | --- |
| **편성** | 몇 명을, 어떤 역할로, 병렬·순차 여부 |
| **작업 예산(task budget)** | **이 작업에 토큰 몇 개까지 — 초과 시 중단**(기능 7 직결) |
| **결과 합성** | 다수결·교차검증·요약 — *"3명 중 2명이 다르게 판정했다"* 를 산출 |
| **실패 처리** | 일부 에이전트 실패 시 부분 결과 진행·재시도·중단 |
| **상관 로깅** | **N개 호출이 하나의 작업에 속함을 기록**(작업 ID) — 없으면 로그가 파편화되어 **감사·비용 분석이 불가** |
| **진행 관측** | *"채점 중 (12/30)"* — 없으면 사용자는 **멈춘 것으로 오해**한다 |
| **폭주 차단** | **최대 반복 횟수 · 타임아웃 · 재귀 깊이 1단계 제한**(하위 에이전트는 다시 팬아웃 못 함) · 재시도 상한 · **서킷 브레이커** — 상세는 기능 7 |

 - **🔴 폭주(runaway) 방지는 오케스트레이션의 필수 요건이다.** 에이전트가 스스로 에이전트를 만들고 도구 호출을 반복하는 구조는 **무한 루프·재귀 폭발의 여지를 구조적으로 갖는다.** *정당한 사용료는 지불하되, **아무도 원하지 않은 호출은 즉시 끊는다*** — **정상 사용의 cap 초과(경고 → 차단)와 비정상 폭주(즉시 중단)는 성격이 다르며, 둘을 섞으면 폭주를 못 막거나 정상 사용까지 끊는다**(기능 7).
 - **팬아웃은 값어치가 확실한 곳에만 쓴다** — 서술형 교차 채점(기능 8) · 위기감지 오탐 검증(기능 5) · 문항 검증·품질 이밸(기능 3) · 깊이조사(기능 1, **사용자가 명시적으로 요청할 때만**). **일반 대화와 자동 백그라운드 작업에는 쓰지 않는다.** 요청 1건이 **API 호출 수백 회**가 될 수 있으므로(에이전트 1명 ≠ 호출 1회), 「학생 1인당 API 사용료 산출」은 **팬아웃 배수를 반영해 재산정**한다 — **연간 벤더 계약 규모의 근거**다(§6).
- **공유 도메인 서비스(중간 계층):** 두 세션 유형(학생/업무)이 함께 쓰는 상태·데이터(학습 상태 · 단일 집계 · 콘텐츠 저장소 · 성적 회송 브로커 · 단일 열람 게이트 · 동의 레지스트리)는 어느 화면에도 속하지 않는 **내부 MCP 계층**으로 둔다 — 코어 / 공유 도메인 서비스 / 화면의 **3계층**이 확정 구조다(§3-6 · 세션정책 §4).
- **모델 = 프로바이더 어댑터:** 코어에는 **공통 인터페이스 + 모델 레지스트리**만 둔다. 새 벤더·모델(Claude·GPT·국산·향후 **교육부 교육특화 LLM**)은 **어댑터 추가 + 설정(config) 등록**으로 붙인다 → **코어 라우터 무수정.**
- **어댑터는 모달리티마다 따로 — 「모달리티별 최적 엔진 조합」:** 어댑터는 대화 모델에만 적용되지 않는다. **① 대화·추론 엔진 ② 이미지 생성·편집 엔진 ③ 음성(STT·TTS) 엔진**이 **각각 독립 어댑터**로 붙는다. 사용자가 대화 모델로 A사를 골라도 「그림공방」(기능 1)은 B사의 이미지 편집 엔진이, 「음성대화」는 자체·제3의 음성 엔진이 수행할 수 있다 — **사용자는 그 조합을 의식할 필요가 없다.**
 - **이는 벤더 앱이 구조적으로 할 수 없는 일이다.** 각 벤더 앱은 자사 엔진만 쓴다(대화가 강한 벤더는 이미지를 못 만들고, 이미지가 강한 벤더는 대화가 약할 수 있다). 바당2만 **분야별 최고를 조합**할 수 있으며, 이것이 벤더 앱 대비 우위의 구조적 근거다.
 - **따라서 「기능 집합은 벤더 무관하게 하나」다.** 사용자가 모델을 바꾸면 바뀌는 것은 **추론 품질뿐**이고, 화면·도구·기능명은 동일하다. 벤더별로 화면을 다르게 만들면 "A사를 고르면 더 많이 된다"를 학습시켜 **모델 선택이 기능 선택으로 변질**되고 어댑터 원칙이 UX에서 무너진다.
- **기능 = 모듈/확장 슬롯 — 붙는 방식은 둘:** 코어 위에 얹는 기능(§4)은 모듈로 추가·교체한다. **사람이 쓰는 교육 도구는 LTI**(LMS 안에 SSO된 채 임베딩), **모델이 쓰는 도구·데이터는 MCP**(Model Context Protocol)로 붙인다 — 새 규격 발명 없이 확장.
- **「새 규격 발명 없이」의 구체 — 확장 패키지 규격은 *발명*이 아니라 *조립*이다:** 확장 패키지(기능 10)는 다섯 층의 묶음이고, **네 층은 이미 국제 표준이 있다.** 표준이 있는 층은 **절대 자체 포맷을 만들지 않는다.**

| 층 | 무엇 | 채택 표준 |
| --- | --- | --- |
| 도구·데이터 연결 | 모델이 쓰는 도구 | **MCP** *(단, 규격 변동 중 — RFP에 버전 고정·변경 통지, §6)* |
| 화면 임베딩 | 사람이 쓰는 도구 | **LTI 1.3 / Advantage** (1EdTech) |
| **문항·평가** | 문항·정답·해설·**루브릭** | **QTI 3.0 프로파일** (1EdTech) — 전면 준수가 아니라 **실제 사용 문항 유형만 규정한 부분집합** |
| **성취기준** | 스킬 맵의 뼈대 | **CASE** (1EdTech) — 국가 교육과정 성취기준 코드를 이 위에 얹는다 |
| 콘텐츠 패키징·메타 | 자료 묶음·메타데이터 | **Common Cartridge · LOM** |
| **공문·문서 산출물** | 공문·가정통신문·보고서 | **hwpx (KS X 6101)** — 공문서 **법정 표준**(2026.10. 의무화) · ODF·PDF/A 병용 |
| **매니페스트** | 이름·버전·작성자·**요구 권한 스코프**·호출 도구 선언·심사 메타 | **표준 없음 — 이 한 겹만 자체 정의**(최소로) |

 - **왜 표준이어야 하는가 — 생태계가 표준 위에서만 자란다.** 문항을 제주 고유 포맷으로 만들면 ① 에듀테크 업체는 이미 가진 QTI 문항을 다시 만들어야 해 **마켓에 오지 않고** ② **AIDT·타 시도 콘텐츠를 수용할 수 없고** ③ 제주가 만든 콘텐츠를 **내보낼 수도 없어** 제주교육의 자산이 제주 안에 갇힌다. 성취기준을 CASE로 표현하면 **교육과정 개정 시 코드 매핑만 교체**하면 된다. 기능 10의 목표(**제주 에듀테크 산업 육성**)는 표준 위에서만 성립한다.
 - **벤더 스킬 형식에 종속되지 않는다.** 벤더의 에이전트 스킬 형식(Anthropic Agent Skills 등)은 **벤더가 주도하는 형식**이고 3사가 제각각이다. **우리 매니페스트를 정본으로 두고 벤더 형식으로는 내보내기(export)만 한다** — 어댑터 원칙 그대로(§3-3). 벤더가 형식을 바꿔도 제주의 스킬 자산은 흔들리지 않는다.
 - **부수 효과:** 구축사가 규격을 **창작할 필요가 없어진다** — 표준을 고르고 프로파일만 정의하면 되므로, 콘텐츠 트랙이 기다리는 「규격 조기 산출」의 리스크가 크게 준다(기능 10).
- **바당2 자신의 기능도 내부 MCP 서버로 구현한다:** 기능 3(학습 상태)·기능 4(학습기록 조회)·RAG(성취기준·자체 콘텐츠 근거 — 코퍼스 권리 단계화는 §3-5)를 **내부 MCP 서버**로 노출한다. 그러면 코어 기능과 서드파티 확장이 **같은 계약** 위에 서고, 바로 아래의 *"확장이 코어 기능의 더 나은 대안이 될 수 있다"* 가 선언이 아니라 구조가 된다. 서드파티에 내미는 SDK의 품질도 **자기가 먼저 써서** 검증된다.
- **코어도 늙는다 — 확장이 코어를 상회 가능(확장판의 존재 이유):** 코어 기능(§4)은 *기본 보장선(baseline)*일 뿐 **만들어진 순간부터 낡는다.** 그래서 **확장 마켓은 코어 기능의 '더 나은 대안'까지 허용**한다 — 코어에 있어도 더 좋은 확장판이 있으면 모듈로 들어와 대체·보완한다(예: 더 정교한 학습기록·생기부 참고자료). **무엇을 쓸지 선택은 학교·교사**가 한다 → 코어가 낡아도 플랫폼은 늙지 않는다.
- **공공은 토대, 민간은 속도 — 역할 분담:** 교육청은 예산·조달·심의 절차상 **빠른 기능 변경이 구조적으로 어렵다.** 반대로 업체는 AI 흐름에 **신속히 대응·갱신**해 응용 기능의 유지보수·최신성에 유리하다. 그래서 **바당2(공공)는 '플랫폼 기능'**(게이트웨이·표준 연계·로깅·안전·거버넌스 = **안정적 토대**)**에 충실**하고, **빠르게 바뀌는 응용 기능은 업체 확장에 맡긴다** — 공공의 느림을 '**안정적 토대 + 민간 속도**'의 분업으로 전환한다.
- **불변 원칙(보안):** 어떤 어댑터·모듈·확장이 붙든 **반드시 코어 게이트웨이를 경유** → 로깅·안전필터·개인정보(PII) 통제를 우회할 수 없다. **MCP 도구 호출·반환 전문도 예외가 아니다** — 도구가 돌려준 데이터는 프롬프트 인젝션·유출 벡터로 취급해 코어 로깅·안전필터를 통과시키고, 도구 권한 스코프는 **호출한 학생·교사의 신원에 종속**시킨다(남의 기록 조회 불가). 외부 확장은 **심사·샌드박싱·권한모델**을 거쳐 동작.
- **도구가 되어선 안 되는 것(금지선):** 인증·모델 라우팅·전량 로깅·PII 비식별·안전필터·위기감지는 **모델이 호출을 선택할 수 있는 도구(MCP)로 만들지 않는다.** 도구는 본질적으로 **모델이 호출 여부를 결정**하므로, 안전 장치를 도구화하면 게이트웨이 독점(§3-1)이 무너진다. 이들은 모델이 건너뛸 수 없는 **파이프라인 구간**으로 강제한다. *(§3-5가 "암호화·재식별은 결정적·검증가능해야 하며 생성형 AI가 아니다"라고 한 것과 같은 이유.)*
- **교사 토글(부차):** 위와 별개로, 일부 기능 모듈은 교사가 학급·과목에 맞게 on/off. 단 **코어·안전·개인정보는 항상 ON.**
- **효과·비전:** 새 모델·기능을 즉시 수용, MVP를 한 번에 다 만들 필요 없이 **점진 확장**. 전면시행 단계에서 심사·인증 거친 **확장 마켓** → 에듀테크 기업·교사 개발자 생태계, **제주 에듀테크 산업 육성**.

### 3-4. 인프라 요약

- 공공기관용 **CSAP 인증 클라우드**(네이버 클라우드[공공]·KT클라우드 등) / 앱서버 2대+피크 오토스케일 / 관리형 **PostgreSQL(HA)** / **Redis** / 오브젝트 스토리지 / 로드밸런서. 부하 기준 **2,200명**(초·중·고 각 2교, 초등은 6학년만).
- OS·DB·캐시 모두 오픈소스. **상세 사양·인프라 비용은 ISMP(~2026-09-30)에서 산정한다**(약식 서버 인프라 초안은 2026.7. 폐기 — 백지 재산정).
- **부하 축은 다섯이다**

| # | 부하 축 | 성격 | 스케일 지표 |
| --- | --- | --- | --- |
| ① | **AI 프록시** (SSE 스트리밍) | 연결 유지형 · CPU 저 | **동시 연결 수** |
| ② | **실시간 채팅** (WebSocket) | 연결 유지형 · CPU 저 | **동시 연결 수** |
| ③ | **비식별 엔진 — 텍스트** | **CPU 집약** · 독립 스케일 그룹 | CPU |
| ④ | **비식별 엔진 — 비전** | **경량 GPU** · 독립 스케일 그룹 (§3-5) | 프레임/초 · GPU |
| ⑤ | **코드 실행 샌드박스** | CPU·메모리 집약 · **강한 격리** (기능 1 「실행실」·「문서공방」·기능 10) | 동시 실행 수 |

- ①·②는 연결을 오래 붙잡되 CPU를 거의 쓰지 않으므로 **오토스케일 지표를 CPU가 아니라 동시 연결 수**로 잡고, 로드밸런서 유휴 타임아웃·응답 버퍼링이 스트리밍을 끊지 않도록 구성한다.
- **CPU 최대 소비처는 게이트웨이가 아니라 비식별 엔진**이다(§3-5) — AI 요청 1건마다 입력·출력 양방향으로 NER 추론이 돌므로 **독립 스케일 그룹**으로 분리한다.
- **GPU 소요의 정확한 범위:** **LLM 추론용 GPU 팜은 여전히 불필요**하다(추론은 외부 API — §3-1 대원칙 유지). 다만 **④ 비전 비식별**(얼굴·이름표 탐지, 라이브 프레임 처리)에는 **경량 비전 가속기(소수)** 가 필요하다 — 규모·성격이 LLM용 GPU와 다르며, 대수·사양은 ISMP에서 산정한다.
- **⑤ 코드 실행 샌드박스는 강한 격리가 요건**이다 — 학생·교사가 짠 코드와 서드파티 확장(기능 10)이 **같은 런타임을 공유**하므로, 컨테이너 격리(경량 VM급)·네트워크 차단·시간/메모리 상한·파일시스템 격리를 RFP 필수 요건으로 둔다.
- **문서 처리는 별도 축이 아니다.** 공문서 법정 표준이 **hwpx**(XML/ZIP 개방 표준·KS X 6101)로 확정되어(2026.10. 의무화 — 기능 1) **일반 리눅스 앱서버·샌드박스(⑤)에서 처리된다.** *(초안 검토 당시 상정했던 「문서 변환 — Windows 인스턴스 + 한컴 라이선스」 축은 **hwp 쓰기가 개통 범위에서 빠지면서 소멸**했다.)*
- **확대 내성(비기능 요건):** 시범 아키텍처는 전면 확대 시 재설계가 아니라 **증설로 가는 구조**(무상태 앱 계층·큐 분리·수평 확장)임을 RFP 비기능 요건으로 명시한다 — 시범 규모와 전면 규모의 단절 방지.
- **백업·재해복구(DR):** 백업 주기와 복구 목표(**RPO/RTO**)를 RFP에 수치로 명시하고 연 1회 복구 훈련을 포함한다. **식별↔익명 매핑 테이블은 별도 백업·복구 절차**로 다룬다(§3-5 최우선 보안자산).

### 3-5. 거버넌스·개인정보

- 식별된 미성년 학생의 대화 전문 = **개인정보이며, 건강·정신건강 등 민감정보(개인정보보호법 §23)가 포함될 수 있는 고위험 데이터**(민감정보 처리는 별도 동의 또는 법령 근거 필요). 벤더에는 **가명처리된 토큰만** 전송 — 매핑 보유로 재식별 가능하므로 법적으로는 익명이 아닌 **가명정보**(§28-4 안전조치 의무 적용)다. 식별 로그는 **교육청 소유·국내 보관.**
- **데이터 최소화(주입 금지) — 비식별의 1차 방어선:** 학생 식별정보는 **애초에 프롬프트·컨텍스트에 넣지 않는다.** 개인화가 필요하면 **가명 토큰과 학습 상태 요약**만 전달하고, RAG 검색 결과·MCP 도구 반환값도 같은 원칙을 따른다. 비식별 엔진은 그럼에도 유입된 PII를 걸러내는 **2차** 장치다. *(§3-1 LAUSD "Ed"의 내부고발 요지가 **"너무 많은 데이터를 보낸다"** 였음을 상기하라 — 문제의 본체는 탐지 실패가 아니라 **필요 이상의 정보를 프롬프트에 주입한 설계**였다.)*
- **기록 ≠ 상시 열람(위축효과 방지):** 전량 로깅의 목적은 안전·감사·환류이지 상시 감시가 아니다. **교사 열람은 집계·요약을 기본**으로 하고 **대화 원문 열람은 필요최소(안전 이벤트·본인 요청 등)·사유 기록**으로 한정하며, 학습 중의 질문·오답·시행착오는 **평가에 불리하게 쓰지 않는다**는 원칙을 이용 안내에 명시한다 — 학생이 안심하고 물을 수 있어야 위기감지·초개별화의 입력이 유지된다.
- **벤더 계약 통제(기술 비식별과 병행):** 모든 LLM 벤더 계약(DPA)에 **① 입력 데이터 학습 미사용 ② 제로 리텐션(또는 최소 보존·즉시 파기) ③ 국외 이전·재위탁 제한 ④ 침해 통지 의무**를 필수 조항으로 명시 — 비식별 엔진의 잔존 누출에 대한 2차 방어선이며, 미제공 벤더는 어댑터 등록 불가. 가명 토큰도 가명정보(§28-4)인 만큼 **국외 리전 처리 시 국외 이전 요건(법 §28-8)의 적법 경로를 PIA 필수 항목으로 확정**하고, 벤더별 기능 대응표에 **국내 리전 제공 여부** 열을 둔다(라우팅 우선순위 반영).

#### 🟦 외부 LLM 전송의 법적 구조 — 다중 검증 조사 결과 반영

> *2026.7. 개인정보보호법·개인정보위 의결례·안내서·국내외 선례에 대한 다중 검증 조사 결과다. **법률 자문(교육청 소속 변호사)·PIA로 확정**하되, 설계 전제는 아래를 따른다.*

**① 법적 성격 — 처리위탁(§26)과 국외이전(§28-8)이 「중첩」된다**

| 축 | 결론 |
| --- | --- |
| **§26 처리위탁** | 실무 통설·규제기관 전제. **단 무조건 성립하지 않는다** — **입력데이터의 학습·재이용 금지가 계약(기업용 API 라이선스·DPA)으로 담보될 때만** 위탁 구성이 선다. **개인용·무료 라이선스는 사업자 자체 목적 이용 우려로 위탁 프레임이 깨지고 「제3자 제공·목적 외 이용」 문제로 전환**된다 → 위 DPA 4개 조항이 **적법성의 전제**이지 부가 조건이 아니다. |
| **§28-8 국외이전** | **중첩 적용된다.** §28-8①은 *"제공·**처리위탁**·보관"* 을 모두 '이전'으로 정의하므로, **위탁이냐 제공이냐의 성격 논쟁과 무관하게 수신자가 해외 사업자면 피할 수 없다.** |

 - **교육청에 불리한 한정:** §28-8①3호(계약 이행에 필요한 위탁)는 *"정보주체와의 계약"* 을 전제하는데, **공교육은 학생과의 사법상 계약이 아니라 법령상 의무 이행**에 가깝다. 미국은 적정성 인정 국가도 아니다 → **교육청의 현실적 근거는 사실상 1호(별도 동의)로 좁아진다.**
 - **🔑 최대 레버 — 국내 리전 처리:** **국내 리전에서 처리하면 §28-8 트리거 자체를 회피**하고 §26 위탁 규율만 남길 수 있다. 이것이 확인되면 **아래 논쟁의 판이 통째로 바뀐다.** **§6의 최우선 확인 항목**이다.

| # | 공백 | 근거 |
| --- | --- | --- |
| **㉠** | **동의는 「적법 근거」를 만들 뿐, 「안전조치」는 별개의 병존 의무다** | §23②는 *"제1항 **각 호에 따라**(별도 동의 포함) 민감정보를 처리하는 경우에도 §29 안전성 확보조치를 하여야 한다"* 고 **문언으로 못 박는다.** §28-8④(국외이전 보호조치)도 동의를 예외 사유로 두지 않는다 |
| **㉡** | **만 14세 미만은 본인 동의가 근거가 되지 못한다** | §22-2①은 **법정대리인 동의 + 동의했는지 「확인(검증)」** 을 **이중으로** 요구한다. 학생이 클릭하는 동의 UI는 확인 요건을 충족하지 못한다 — **초등학생 전원 + 중1 일부**가 여기 걸린다 |
| **㉢** | **프롬프트에 등장하는 제3자는 동의 주체가 아니다** | 교사가 *"학생 A의 상담 내용"* 을, 학생이 *"친구 B의 이름"* 을 쓰는 순간 **이용자 동의 스킴은 그 데이터에 아무 근거도 제공하지 않는다.** **동의로는 원리적으로 메울 수 없고, 입력 필터(비식별)+이용정책+교육으로만 통제된다** — **마스킹 해제를 반박하는 가장 강력한 단일 논거** |
| **㉣** | **포괄·약관형 동의는 무효에 가깝다** | 목적별·항목별로 구분된 **명시적 별도 동의**가 요구된다(이루다 2021 · 비대면진료 의결 §22① 위반) |

**③ 마스킹(비식별)의 법적 지위 — 정확히 쓴다**

> **⚠️ 「마스킹 = 법정 의무」는 성립하지 않는다.** 조문상 **암호화는 의무형이나 마스킹(표시제한)은 "…할 수 있다"는 허용형**이다. 이를 과장하면 역풍을 맞는다.
>
> **✅ 방어 가능한 명제는 이것이다:** **동의는 안전조치 의무를 면제하지 못하며**(§23②→§29 · §28-8④), **마스킹은 그 안전조치를 구현하는 「채택된 수단」**이므로, **이를 해제하려면 위험도 기반 정당화 · 대체 통제 제시 · PIA 기재 부담을 진다.**
>
> **근거:** 개인정보보호위원회 **「생성형 AI 개발·활용을 위한 개인정보 처리 안내서」(2025.8)** 는 **「모델이용자」**(외부 상용 모델을 API 연계하는 기관 — **바당2가 정확히 해당**)의 시스템 안전조치 이행항목으로 **「입출력 필터」를 명시적으로 열거**한다. 안내서는 비구속 행정지도이나 **스스로를 "최소한의 사항"으로 규정**하고 같은 위원회가 의결·실태점검으로 집행하므로 **사실상의 심사 기준**으로 작동한다 — **「법적 의무」가 아니라 「규제기관의 기대 수준」으로 정확히 표기한다.**

**④ 비식별 처리 방식 — 「기본 On + 계층적 봉쇄 + 조건부 해제 + 전량 감사」**

| 모드 | 동작 | 기본값 |
| --- | --- | --- |
| **Mask** | 가명 토큰으로 치환해 전송 | **학생 세션 기본** |
| **Mask + 재식별** | 가명으로 보내고 **출력단에서 실명 복원**(KMS/HSM·매핑 룩업) — **교사는 실명 문서를 받고, 벤더는 실명을 본 적이 없다** | **업무 세션 기본** |
| **Block** | 민감정보 포함 시 질문 차단 | 고위험 업무 유형 |
| **해제(Off)** | 실명 그대로 전송 | **아래 조건 전부 충족 시에만** |

 - **🔒 (1) 기본값 On · Fail-closed** — 비식별은 **코어 게이트웨이에서 강제**되며, **필터 장애 시 전송을 차단**한다(§3-5 스트리밍 게이트 degrade와 동일 원칙).
 - **🔒 (2) 해제 불가 구간 — 어떤 승인으로도 열 수 없다**

| 구간 | 근거 |
| --- | --- |
| **만 14세 미만 학생의 정보** | §22-2 법정대리인 동의·확인 미충족 시 |
| **민감정보** — 상담·정신건강·건강·**위기신호** | §23 |
| **고유식별정보·주민등록번호·계좌·카드번호** | |
| **프롬프트에 등장하는 제3자의 식별정보** | **이용자 전원 동의로도 원리적으로 메울 수 없다**(㉢) |

 - **(3) 해제 가능 조건(전부 충족)** — ① 해당 처리행위에 **§28-8①의 적법 근거가 개별 확보**(별도 동의 + §28-8② 5개 사항 사전 고지, **또는 국내 리전 처리로 §28-8 미해당 구성**) ② **§26 위탁계약·DPA**에 학습·재이용 금지·재위탁 통제·파기·감사권 명시(기업용 API 라이선스) ③ **처리방침에 국외이전 고지사항 누락 없이 기재**.
 - **(4) 해제 단위 — 전역 토글을 두지 않는다.** 목적·기간·데이터 항목·대상 사용자군을 **특정한 요청 단위**로만 가능하며 **기본 만료(TTL)** 를 둔다.
 - **(5) 거버넌스** — 해제는 **개인정보보호책임자(CPO) 승인 사항**. 사유·승인자·범위·기간을 기록하고, **해제 상태의 모든 요청은 별도 로그 계열로 전량 감사·정기 보고**(§29 접속기록).
 - **(6) PIA 등재** — **해제 기능의 존재 자체**를 침해요인 분석에 기재하고(프롬프트 인젝션·학습데이터 복원과 함께), **잔여위험과 대체 통제**(암호화·접근통제·전송 후 파기 확약·리전 고정)를 명시한다.

> **⚠️ 문서·RFP에서 쓰지 말아야 할 표현 셋** — ❌ *"마스킹은 법정 의무다"*(조문 근거 없음) · ❌ *"동의를 받으면 적법하다"*(§22-2·§23·제3자·안전조치가 남는다) · ❌ *"개인정보위가 마스킹 해제를 금지했다"*(그런 처분은 없다).
>
> **➡️ 설계 목표 — 「해제가 필요 없게 만든다」.** 「Mask + 재식별」이 있으면 **교사는 실명 문서를 얻고 벤더는 실명을 보지 않는다** — 해제를 원하는 상황의 대부분은 **재식별 미구현의 증상**이다. **명부에 없는 사람**(학부모·외부 강사·민원인)은 **세션 스코프 임시 매핑**(세션 종료 시 파기)으로 처리한다.
- **식별↔익명 매핑 격리(최우선 보안자산):** 익명 토큰과 식별정보를 잇는 **매핑 테이블 = 단일 재식별 통로**다. **KMS/HSM 기반 키 관리·별도 보안 도메인 격리**, 식별 로그 접근은 **2인 승인·전체 감사로그**로 강제하고, PIA에서 **최우선 위험자산**으로 다룬다.
- **구현 방식(암호화 ≠ AI):** 두 작업을 분리한다. **① 암호화·복호화·재식별 = 암호기술(KMS/HSM)·매핑 테이블 룩업** — *생성형 AI가 아니다(결정적·검증가능해야 함).* **② PII 탐지·비식별(지능 필요) = 규칙(정규식: 주민번호·전화·이메일) + 경량 NER(한국어 개체명 인식) 기반 온프레미스 비식별 엔진**(오픈소스, 예: Presidio + KLUE/KoBERT급 소형 NER). 결정적·재현가능·감사가능하고 **텍스트 축은 CPU급으로 동작한다.** 풀 생성형 LLM을 비식별에 쓰는 것은 과하고(느림·고비용·환각·검증난) 권장하지 않는다.
 - *(맥락 의존 간접식별자 — 위험의 위치)* 별명·정황 결합("우리 반 반장")은 **읽는 쪽이 학급 명부를 가질 때만** 재식별된다. 벤더는 명부가 없으므로 이 위험은 **교실 내 시연**(기능 1 '대화 공유·시연' · 기능 2 '학생 산출물 시연')과 **교사의 원문 열람**에서 발생하며, 그쪽의 **건별 동의·익명 기본값·열람 최소화**로 다룬다(기능 9 수업 나눔은 교사 저작물만 공유하므로 해당 없음). 벤더 전송 경로의 탐지율을 더 높이려면 **온프레미스 소형 LLM**을 보조 필터로 얹을 수 있으나 — **코어가 아닌 선택적 고도화**이며 GPU·운영비 가산을 동반한다.

- **비식별 엔진 — 비전 축:** 비식별은 더 이상 텍스트만의 문제가 아니다. 기능 1의 개통 필수 목록이 **이미지·영상 축을 다섯 경로**로 연다 — ① **학생 이미지 입력**(손글씨 풀이·사진) ② **pptx·문서 내 삽입 사진** ③ **「그림공방」 생성** ④ **「그림공방」 편집**(입력 이미지 필수) ⑤ **「라이브튜터」**(카메라·화면 공유의 실시간 프레임). **이미지 속 얼굴·이름표·손글씨 이름은 텍스트 비식별로 걸러지지 않으므로**, 코어에 **비전 비식별**을 정식 축으로 신설한다.
 - **구성:** **얼굴 탐지·마스킹** + **이름표·손글씨 OCR 후 텍스트 비식별 엔진 재투입** + **EXIF 제거**. 라이브 경로는 **프레임 샘플링 후 동일 파이프라인**을 통과시킨다.
 - **비용:** 실시간 프레임 처리 때문에 **경량 비전 가속기(소수)** 가 필요하다 — **LLM 추론용 GPU 팜은 여전히 불필요**하다(§3-1·§3-4 ④).
 - **검수:** 얼굴 탐지 **재현율 하한·오탐 상한·프레임 처리 지연**을 수치로 확정하고, **평가셋은 교육청이 제작**해 RFP에 첨부한다(텍스트 비식별과 동일 원칙 — §6).
 - **이로써 §6의 미결 항목 「학생 이미지 입력을 열 것인가」는 해소된다 — 연다.** 단 **비전 비식별을 반드시 경유**하며, "전면 개방-무처리" 조합은 배제한다.

- **새 위험 축 — 딥페이크(생성·편집물의 오남용):** 비식별이 *"학생 정보가 새어나가는"* 방향을 다룬다면, 「그림공방」의 **이미지 편집**은 **"학생 정보로 무엇을 만드는가"** 라는 반대 방향의 축을 연다 — 최신 이미지 편집 모델은 **인물 일관성을 유지한 채 대화형으로 편집**하므로, 얼굴 사진 한 장이 곧 학교폭력 도구가 될 수 있다.
 - **① 얼굴 포함 이미지는 편집 도구에 투입할 수 없다 — 예외 없음.** 이미지 편집은 입력 이미지가 있어야 성립하므로 **입력 필터가 유일한 방어선**이다(학생 세션·업무 세션 공통). *(교사의 정당한 인물 사진 편집 수요가 있다면 별도 승인·감사 경로로 다루며, 기본 기능으로 열지 않는다.)*
 - **② 생성·편집물 전량 로깅 + 출처 표식(워터마크)** — 누가 무엇을 만들었는지 추적 가능해야 한다.
 - **③ 연령별 유해성 필터를 프롬프트·결과 양방향에 적용**한다(기능 5).
- **스트리밍 하 출력측 안전·비식별(표시 게이트/탐지 경로 분리):** 스트리밍 UX(기능 1)와 출력 방향 필터는 설계상 긴장 관계다 — 다음 구조로 해소한다. ① **표시 게이트(동기·저지연):** 응답 스트림을 문장/절 단위 세그먼트로 마이크로배칭하고, 직전 세그먼트 꼬리를 포함하는 **중첩 검사 윈도**(최장 PII 패턴 길이 이상)로 정규식+경량 NER·유해 카테고리 필터를 통과한 세그먼트만 표시한다(청크 경계에 걸친 이름·연락처 대응). ② **탐지 경로(비동기):** 위기감지·정밀 재검사·감사 적재는 **턴 완료 후 전문 기준**으로 수행한다 — 위기 신호는 문장이 아니라 대화 맥락에서 뜨므로 표시 차단이 아니라 알림 경로이며, 이를 동기 게이트에서 빼는 것이 상충 해소의 본체다. ③ 비동기 검사에서 누출 발견 시 **소급 마스킹**(표시 세그먼트 치환)을 지원하되 보증선은 어디까지나 동기 게이트다(안전은 서버 파이프라인 — §3-3 금지선). ④ 동기 게이트 장애 시 스트리밍을 중단하고 완성 후 일괄 표시로 degrade(안전 우선). ⑤ **검수(수치 2종 동시 충족):** 출력 경로 PII 재현율(교육청 제작 평가셋)과 체감 지연(첫 세그먼트 표시·세그먼트 간 지연의 p95 — 전체 요청의 95%가 그 이내여야 하는 값)을 RFP 검수 시나리오로 명시한다(목표 수치는 §6 — ISMP 확정). 두 수치를 함께 걸지 않으면 구축사는 필터를 끄거나(안전 구멍) 스트리밍을 포기한다(우회 유발).
- **RAG 근거 코퍼스 — 권리 3층 단계화(저작권):** ① **시범(기본):** 공공저작물(국가 교육과정·성취기준 고시)과 **자체 제작 자료**(스킬 맵·문항·해설·루브릭, 이용 허락 범위를 명시한 교사 저작물) — 교과서 원문 없이도 근거 접붙임(§4 품질 3종 ⓑ)은 성립하며, 기능 3 콘텐츠 트랙이 곧 코퍼스 제작 트랙이다. ② **확장(라이선스 확보 시):** 교과서·상용 콘텐츠 — 발행사 라이선스는 ISMP에서 비용·협상 트랙으로 별도 타진하되 **시범 필수 경로에서 제외**한다(권리 협상이 개통을 인질로 잡지 않도록). ③ **금지:** 권리 미확인 자료의 코퍼스 적재 — 교사 업로드 자료는 **해당 수업 맥락 한정 사용**(코퍼스 승격 금지). 코퍼스 등재는 **출처·권리 메타데이터 필수**(콘텐츠 저장소 스키마) + 등재 심사(확장 마켓 심사와 동일 원칙)로 하고, 튜터 응답에는 **근거 출처를 표기**하며, 권리 만료 시 제거 파이프라인을 RFP 요건에 포함한다.
- **로그 무결성:** 전문 로그·감사로그는 **변조 방지 저장**(추가 전용·해시 체인 등 무결성 검증 가능 방식)과 시각 동기화(NTP)를 RFP 요건으로 한다 — 전량 로깅이 감사·분쟁의 증거가 되기 위한 전제.
- 사전 정립 필요: 보관기간 · 접근권한(감사로그) · 이용목적 제한 · **개인정보 영향평가(PIA)** · 개인정보보호법 자문(별도).

### 3-6. 세션 정책 (요약)

> **정본(기준서)은 「세션정책」(내부 문서)**(2026.7.11. 다관점 교차 검토로 확정)이다 — 본 절은 그 구조 요약이며, **상충 시 세션정책이 우선**한다. ISMP·RFP·상세 설계·PIA·검수 기준은 세션 관련 사항을 세션정책에서 인용한다.

- **세션 = 정책 적용 단위:** 보호·권한·감사 규칙은 앱·화면·배포가 아니라 **세션**(인증된 사용자들의 상호작용 묶음)에 붙는다 — **정책은 세션을 따라가고, 세션은 사람(role)을 따라간다.**
- **세션 유형 2종 — role 기준 판정:** 「세션에 role=학생이 참여하는가」로 코어가 분류한다. **학생 세션**(학생이 참여하는 세션 전부 — 연령 필터·위기감지·동의·기록 전면)과 **업무 세션**(교직원 단독 작업 — 열람·갱신 감사 중심). 인증 프로토콜(LTI/SAML)·시간·장소는 판별자가 아니다 — 학생의 심야 자율 학습도 학생 세션이다.
- **활동 맥락 태그(수업/자율):** 학생 세션 안의 보조 태그 — **정책을 바꾸지 않는다**(보호는 맥락 무관 동일). 용도는 기록 분류·열람 세분·토큰 한도. 표기 축약: 「학생 세션(수업)」·「학생 세션(자율)」.
- **3계층 구조:** ① **코어 게이트웨이**(하나·불변 — 인가 서버 겸임: 세션 분류·사용자 귀속 토큰 발급·정책 집행) ② **공유 도메인 서비스**(내부 MCP — 두 세션 유형이 함께 쓰는 상태의 단일 원본, §3-3) ③ **화면 계층**(수업 화면 = 학생 세션 담당·LTI 등록 / 업무 화면 = 업무 세션 담당·SAML 직접).
- **협상 불가 4종(RFP 필수 요건):** ① 코어 = 인가 서버 — actor=학생 이벤트는 사용자 귀속 토큰만 수용하고 actor==token.sub를 서버에서 검증 ② 화면 계층의 학습기록·상태 DB 직접 접근 금지 ③ 학생 기록 열람은 **단일 열람 게이트**만 경유(집계·요약 기본 — 원문은 별도 스코프+사유+승인+전건 감사) ④ 안전·비식별·로깅은 전적으로 서버 파이프라인(클라이언트 측 배제).
- **검수 가능 요건 4시험:** 배포 독립성(트랙 B 배포 중 학생 세션 무중단) · 행위자 무결성(업무 세션에서 학생-actor 이벤트 0건) · 게이트웨이 경유율(우회 경로 부재 100%) · 경계 관통 인터페이스(채점 확정→회송 등 데이터 계약 검수).
- **릴리스 2트랙:** 트랙 A(수업 화면 — 배포 창 제한·안전 이밸 게이트) / 트랙 B(업무 화면 — 상시 배포). 그럼에도 사용자에게 바당2는 **하나의 앱**이다 — 재인증 0회·공통 셸·딥링크 승계, 학생에게는 구분 자체가 보이지 않는다.

## 4. 바당2 주요 기능 (기능 모듈)

> 코어 게이트웨이(§3-1·§3-3) 위에 얹는 **기능 모듈**(대체로 동급). 단 **벤더 앱급 AI 웹 클라이언트(기능 1)은 다른 기능이 올라타는 공통 웹 클라이언트(토대)를 겸하며** — 실시간 채팅 수업 도구(기능 2)도 그 위에서 동작한다. 앞쪽은 **토대·사용 빈도 높은 주요 기능**(AI 접근·실시간 수업·학습 루프·기록·안전·개인정보·형평·비용), 뒤로 갈수록 교육 활용·운영·확장. 도입 범위·우선순위는 RFP 단계에서 확정.

> **초개별화 작동 원리 — 이 기능들이 함께 굴리는 하나의 루프.** 아래 기능들은 흩어진 도구가 아니라, 블룸 **완전학습(Mastery Learning)**의 **「숙달을 향한 목표지향 루프」(이하 완전학습 루프)를 학생마다 실시간으로 돌리는** 하나의 엔진을 이룬다.
> - **학생별 튜터 에이전트(루프·스킬):** 학생마다 하나의 AI 튜터가 설명·발문·힌트·재출제 같은 **스킬**을 골라 쓰며 *숙달*을 향해 대화를 이어간다(기능 1 위에서 작동 — **기능 3으로 구현**).
> - **하네스 = 코어 게이트웨이:** 이 에이전트들을 **안전하게 구동·기억·기록·환류**하는 공통 실행 틀. 모든 상호작용이 관문을 지나 로깅·안전·PII 통제를 우회할 수 없다(§3-1·§3-3). 전량 기록은 학습기록·성장 이력에 적재(기능 4).
> - **교사 = 지휘자:** 학습목표·성취기준·안전 경계를 정하고 각 학생·에이전트의 진행을 모니터링하며 개입 지점을 결정(기능 1·2·8, 현황판은 기능 3). 개별화는 학생 한 명 한 명 층위에서, 통합은 교사가.
> - **품질 3종('잘 설계된 AI'의 실체):** ⓐ **지식 추적** ⓑ **검색증강(RAG)** — 성취기준·자체 콘텐츠(스킬 맵·문항·해설)에 근거를 접붙여 환각을 억제(콘텐츠 생산이 아니라 근거 접붙임 · 코퍼스 권리 단계화는 §3-5) ⓒ **이밸(evals)** — ⓐ·ⓒ(상태·계측)의 구현은 **기능 3**(학습 상태·루프 인프라) 참조.
>
> *(이론적 뿌리: 「블룸 2σ 이론적 배경」. 블룸↔에이전트↔바당2: 숙달=목표, 형성평가·교정=루프·스킬, 결과 데이터=환류.)*

> **통합 입력 모델 (기능 1·2 공통 토대):** 학생·교사는 **채팅 입력창 하나**를 쓰고 **보낼 대상**(AI · 동료 학생 · 교사 · 학급/모둠)만 고른다 — AI는 대화 상대 중 하나. 단 대상에 따라 메시지의 *모양*이 다르다.
> - **공통(통합):** 입력창·전송·봉투(보낸/받는 대상·스레드·로깅·안전필터)는 하나 — 모든 메시지가 대상 무관하게 **코어**를 거친다.
> - **분기(본문 타입):** 사람 간은 텍스트·파일(단발), AI는 스트리밍·구조화 출력(마크다운·코드·표·이미지·산출물·도구호출, 맥락 유지) → **콘텐츠 타입별 렌더러**로 처리.
> - **따라서:** 입력·전송 토대는 기능 1·2가 공유하고, **AI의 다양한 출력 프로토콜·렌더링이 기능 1의 핵심 난이도이자 값어치**다.

### 기능 1 — 벤더 앱급 AI 웹 클라이언트 (공통 토대·상시 추격)

**모든 AI 대화가 지나는 공통 클라이언트이자, 통합 입력 모델(§4 서두)의 웹 클라이언트 토대**(기능 2의 사람 간 채팅도 이 클라이언트 위에서 동작한다). 게이트웨이는 단순 텍스트 프록시가 아니라 **벤더 공식 앱(ChatGPT·Claude·Gemini 등)에 준하는 사용 경험**을 제공해야 한다 — 품질이 낮으면 학생이 벤더 앱으로 **우회**해 전량 로깅(게이트웨이 독점)이 무너지므로, **UI/UX 품질이 사업 성패의 전제**다. 통합 입력창(위)에서 대상이 **AI**일 때 작동하며, 사람 메시지(텍스트·파일)와 달리 **다형·스트리밍 출력을 콘텐츠 타입별 렌더러로 처리**하는 것 — 이 출력 프로토콜·렌더링이 본 기능의 핵심 난이도이자 값어치다.

> **반면교사 — 'API만 붙인' AI 서비스의 전형적 실패 패턴:** ① 스트리밍 없음(응답 완성까지 빈 화면 → 이탈) ② 마크다운·표·수식·코드 렌더링 부실 ③ 대화 맥락 관리 실패(매번 처음부터 설명) ④ 결과물 활용 불가(복사·다운로드·재편집 안 됨) ⑤ 모바일 미대응. **모델 성능은 API로 평준화되므로 승부는 인터페이스**다 — 교육용 AI 서비스가 파일럿만 하고 사라지는 주된 이유.

**핵심 요건** — 위 다섯 실패 패턴의 역상 + 교육 활용 요건:

- **스트리밍 응답**(출력측 안전·비식별과의 양립 방식은 §3-5 「표시 게이트/탐지 경로 분리」) · **대화 세션·이어가기·히스토리**(맥락 유지) — **컨텍스트 정책**(히스토리 절단·요약, 학습 상태 주입 시점, 시스템 프롬프트·자료의 캐시 적중 설계)을 RFP 제안 필수 항목으로 요구하고, 비용 검수는 **차시당 빌링 토큰**으로 수치화한다(「학생 1인당 API 사용료 산출」과 정합)
- **렌더링**: 마크다운·표·수식·코드
- **멀티모달**: **입력**은 이미지·문서·파일 첨부(음성은 STT 경유) · **출력**은 아래 「모달리티 사실」 참조
- **결과물 활용(후속 작업)**: 복사·다운로드·파일 저장·재편집 — AI 산출물이 **수업 자료의 생성·편집·배포로 이어지는 흐름**이 실제 정착률을 결정한다(기능 8 작업대·기능 9 수업 나눔 연계)
- **모바일·태블릿 대응**: 반응형 UI — 학생 사용 환경의 다수가 모바일임을 전제
- **모델 선택**, 프롬프트 템플릿, 즐겨찾기·공유
- **대화 공유·시연**: 특정 학생의 AI 송수신(멀티모달 Q&A)을 학급에 공유·시연 — 단 **당사자 학생의 사전 동의(건별)·거부권**을 UI로 강제하고, **익명(이름 가림) 공유를 기본값**으로, 공유 전 교사·학생이 미리보기로 확인 (수업 중 활용은 기능 2)

> **⚠️ 모달리티 사실 — 오해 방지:** LLM API는 **입력이 텍스트·이미지·문서까지 멀티모달이나, 음성 입력은 없고 출력은 텍스트 전용**이다(text · 사고 · 도구호출 블록). 마크다운·표·수식·코드의 "풍부한 응답"은 **클라이언트 렌더링**이고, **그림·음성·문서 파일은 도구 계층이 만든다.**
> **따라서 기능 1의 값어치는 *모델이 주는 것*이 아니라 *우리가 얹는 것*에 있다** — 벤더 앱과 동급의 경험은 API를 붙여서 얻어지지 않고, 아래 목록을 **직접 만들어서** 얻어진다.

> **🟦 hwpx가 정본인 이유 — 정부가 「AI가 읽는 행정」으로 방향을 정했다.** 「행정업무 운영 및 혁신에 관한 규정」 개정안이 **2026.5.12. 국무회의 의결**되어 **2026.10.부터 공공기관 기본 문서 포맷이 hwpx로 의무화**된다(온나라 문서시스템은 5.18.부터 개방형 문서만 첨부 허용 · ODF·PDF/A 병용). **전환 이유가 곧 "AI가 읽기 좋은 구조"** 다 — hwpx는 XML/ZIP 개방 표준(**KS X 6101**)이라 표·조항 구조가 기계 판독된다.
> - **정부가 「AI가 읽는 행정」을 위해 hwpx를 의무화했고, 바당2가 바로 그 AI다.** 정책 방향과 사업이 정면으로 일치한다.
> - **따라서 hwp(구 바이너리) 쓰기는 개통 범위에서 제외한다.** 개통(2028-09)은 의무화 후 **약 2년** 시점이라 교육청 공문은 전부 hwpx다. hwp는 **축적된 레거시 자료를 읽기 위해서만** 지원한다 *(레거시 hwp 생성이 필요해지면 확장으로 붙인다)*.
> - **부수 효과:** ① **한컴 서버 SDK·Windows 변환 팜이 불필요**해져 라이선스 협상 트랙이 소멸한다(§6) ② hwpx는 일반 앱서버에서 처리되므로 **인프라 부하 축이 늘지 않는다**(§3-4) ③ 2028년이면 교육청 규정·지침이 전부 hwpx라 **「규정·지침 질의」 스킬의 RAG 코퍼스 품질이 근본적으로 좋아진다**(기능 10).
> - **「한글 편집기」를 개통 필수로 두는 근거도 여기 있다** — hwpx는 개방 표준이라 **브라우저 편집기를 자체 구현할 수 있고**, 공문은 표·서명란 등 **서식이 본질**이라 리치텍스트 절충으로는 업무가 완결되지 않는다. **AI가 초안을 만들고 → 브라우저에서 서식 그대로 고치고 → 그대로 결재에 올린다.**

**설계 판단 근거 — 왜 이렇게 나눴는가**

- **「바로분석」 vs 「실행실」 (2층 실행 모델):** 브라우저 실행은 **데이터가 단말을 떠나지 않고 서버 비용이 0**이라 탐색적 학습에 최적이나, **학생이 결과를 위조할 수 있다.** 따라서 **평가·제출·기록에 반영되는 실행은 반드시 서버층(실행실)** 으로 강제한다. 저사양 단말은 서버층으로 자동 폴백한다(기능 6 형평성).
 - *(§3-6의 "안전·비식별·로깅은 전적으로 서버 파이프라인, 클라이언트 측 배제" 원칙과 **충돌하지 않는다.** 그 원칙은 **안전 판단**을 클라이언트에 두지 말라는 뜻이고, 브라우저 실행 결과는 다음 턴에 **코어를 거쳐** 모델로 들어가므로 로깅·필터가 그대로 성립한다.)*
- **「문서공방」·「한글 편집기」 — 벤더가 절대 못 하는 영역:** 공문·가정통신문·기안의 법정 표준은 **hwpx**이며, 벤더 앱은 읽지도 만들지도 편집하지도 못하고 **앞으로도 만들지 않는다**(한국 공공 수요는 그들의 로드맵에 없다). **AI가 초안을 줘도 결국 한글 프로그램에 옮겨 적어야 한다면 업무는 절반만 자동화된 것**이다 — **생성·편집·저장이 브라우저에서 완결**되어야 §0의 「벤더 구독 재가입 최소화」가 성립한다.
- **「학사·급식 도우미」 — 행정 시스템 연동의 첫 실증 사례:** 학사·급식 일정은 교사가 **매일** 쓰는 정보다. **NEIS 교육정보 개방 API**가 공개되어 있으므로 **내부 MCP로 붙이는 정공법**이 그대로 성립한다 — *"행정 시스템 자동화의 정공법은 정식 API 연동(MCP)이고, 화면 조작(「업무자동화」)은 그것이 없을 때의 폴백"* 이라는 원칙의 **첫 실증**이다.
- **「문서함」 — AI 산출물은 어디에 쌓이는가:** 「지식창고」는 **입력** 자료 저장소다. **AI가 만든 산출물**(공문·가정통신문·활동지·문항)이 쌓이는 곳이 필요하다. 문서함은 산출물을 **보관·검색·필터**하고 **공개 범위(전체/우리학교/나만)** 를 지정한다 — 「나만의 AI」·「수업 나눔」과 **같은 3단 공개 모델**을 쓴다.
- **「문서공방」 — 학교 공식 양식 등록:** 학교마다 공문·가정통신문 양식이 다르다. **학교 hwpx 양식을 등록하고 `{제목}`·`{인사말}` 같은 출력 위치를 지정**해 두면, AI가 만든 내용이 **그 학교 공식 서식에 자동 배치**된다. 실무에서는 **학교별 양식 등록**이 핵심이다.
- **「나만의 AI」 — 심사 없이도 안전한 이유:** 교사가 시스템 프롬프트·지식·도구를 조합해 만든 AI도 **코어 안전필터·비식별·전량 로깅을 우회하지 못한다**(§3-3 불변 원칙). **심사는 *코드*를 볼 때 필요한 것이지 *프롬프트*를 볼 때 필요한 것이 아니다.** 따라서 자기 학급 배포는 심사 없이 허용하고, 동료 공유(기능 9)·마켓 게시(기능 10, 2단계 심사)로 올라가는 **3단 사다리**를 둔다 — 이 0단이 없으면 「확장팩」·「수업 나눔」·「확장 마켓」이 전부 **교사가 닿을 수 없는 상층부**만 남는다(기능 10).
- **「업무자동화」는 폴백이지 정공법이 아니다:** 행정 시스템 자동화의 정공법은 **정식 API 연동(내부 MCP)** 이다. 화면 조작은 API가 없을 때의 최후 수단이며, **화면 캡처가 곧 학생 개인정보 화면**이므로(나이스 등) **화면 비식별(§3-5 비전 축)과 반드시 함께** 개발한다. **학생 세션에는 이 기능이 아예 노출되지 않는다.**
- **「AI 선택」은 학생에게 열지 않는다:** 모델마다 비용·안전 등급·기능 지원 범위가 다르다. **교직원은 자유 선택, 학생 세션은 교육청 화이트리스트**(또는 교사가 차시별 지정)로 관리한다 — 토큰 cap(기능 7)·연령별 안전필터(기능 5)와 정합한다. **DPA 4개 조항(§3-5)을 충족하지 못하는 모델은 애초에 어댑터 등록 불가**다.

**하지 않는 것 — 기능 결핍이 아니라 정체성**

| **벤더 서버 도구 직결** | 기능은 자체 구현으로 대체하되, *벤더 것 그대로*는 불가 — 로깅·비식별·국내보관이 끊긴다(§3-1 두 번째 금지선) |
| **모델 파인튜닝·커스텀 모델** | **벤더가 API로 열지 않는다** — 우리 의지의 문제가 아니다. 교육특화는 KERIS 별도 트랙이거나 프롬프트·RAG로 |
| **영상 생성**(Sora·Veo급) | 개통 범위 밖 — **확장 슬롯**(기능 10)으로 열어두되 코어가 막지 않게 한다. 비용·수요 확인 후 결정 |

**진화 지속성 — 벤더 앱 추격 체제(기능 1의 관건)**

기능 1의 성패는 **둘 다**에 달렸다 — **① 개통 시점에 위 「개통 필수 목록」을 실제로 갖추는가** ② 그 이후 **상용 서비스(ChatGPT·Claude·Gemini)의 변화를 계속 따라가는가.** **①이 없으면 ②는 성립하지 않는다** — 개통 후에는 예산이 없다. 월 주기 추격은 **개통 필수 기능의 미구현을 정당화하는 근거가 아니라, 개통 *이후*의 최신성 유지 메커니즘**이다.

> **모델 최신성과 기능 최신성은 분리해 요구한다(검수 가능한 형태):** ① **모델** — *"개통 시점 기준 각 벤더의 최신 GA 모델을 어댑터로 지원"*(어댑터 방식이라 코어 무수정 — 개통일에 맞출 수 있다). ② **기능** — *"벤더 앱 기능 격차 점검표를 분기 갱신하고, 채택 결정 항목을 다음 월 릴리스에 반영"*. **"오픈 시점 최근 1주일 이내 벤더 신기능 제공" 같은 요건은 RFP에 쓰지 않는다** — 검수가 불가능하기 때문이다(§6 "수치 없는 요건은 인수 검수가 불가능하다").

- **기반 선택**: 바닥부터 짓지 않는다 — **검증된 오픈소스 챗 클라이언트 기반** 채택을 우선 검토(커뮤니티가 벤더 앱 수준 UX를 지속 이식·갱신). 바당2 고유부(게이트웨이 경유·안전필터·LTI 결합·현황판)만 자체 개발로 얹는다. 채택 전제 둘을 RFP에 명시: ① **라이선스 실사** — 후보 클라이언트의 라이선스(카피레프트 전파·브랜딩·상업 조항)와 고유부 소스 공개 의무를 법률 검토로 확정하고, 라이선스 변경·프로젝트 중단에 대비해 **후보 2종 이상 비교·전환(exit) 경로**를 둔다 ② **고유부 격리** — 클라이언트 본체를 고치지 않고 **플러그인·확장 지점·프록시 계층**에 얹어(패치 최소화) 업스트림 추종과 커스텀이 충돌하지 않게 한다. 후보 2종 이상의 **라이선스 실사 초안은 ISMP 단계(~2026-09-30)에 완료**한다 — FP 재사용 감액의 근거를 겸한다.
- **상시 개선 체제(운영 주체)**: 일회성 구축 + 연 단위 유지보수로는 추격 불가 → **제주AI미래교육원 「에듀테크소프트랩」** 소속 **교육연구사들이 개발 업체와 함께 전담 지원**하는 상설 체제 — 현장 요구 수집 → 우선순위 결정 → 개선 반영의 상설 루프. 월 주기 릴리스는 **구축 계약과 분리된 「지속 개선(고도화)」 계약**으로 발주하고(**투입공수(M/M) 기반 고도화 대가**로 산정 — 월 주기 상시 개선은 유지관리 요율제로 산정 불가, 세션정책 §11-2), **스테이징 환경·자동 회귀 테스트·기능 플래그·원클릭 롤백**을 구축 필수 산출물로 요구하며, 배포는 **수업 외 시간대 창(window)** 으로 제한한다 — 안전필터·비식별 등 코어 안전부 변경은 릴리스 주기와 별도의 검증 게이트를 거친다.
- **월 주기 개선 = 차별화 전략**: 코어·클라이언트를 **최소 한 달 주기로 개선·릴리스**해 상용 서비스에 준하는 최신성을 유지 — 이탈(우회)을 막는 **이 플랫폼의 차별화 전략 중 하나**다. 벤더 앱 대비 기능 격차를 상시 점검해 다음 릴리스에 반영한다(릴리스 주기는 RFP·SLA에 명시). 단 **실시간 음성은 텍스트와 통제 지점이 다르다** — **벤더 직결형 실시간 음성 API는 게이트웨이를 우회하므로 사용하지 않는다**(§3-1 두 번째 금지선). 대신 **자체 스트리밍 STT → 텍스트(비식별·안전필터 경유) → TTS** 스택으로 「음성대화」를 **실시간 대화급으로 구현**한다(기능 6) — *"벤더가 안 주니 저품질로 참는다"가 아니라 "우리 스택으로 동급 지연을 낸다"* 가 목표이며, 지연 목표치는 §6·ISMP에서 수치로 확정한다. ※ §3-3 "코어 수정 없이 확장"은 *확장·새 모델이 코어 수정을 요구하지 않는다*는 구조 원칙이며, 코어 자체의 지속 개선과 모순되지 않는다.

### 기능 2 — 실시간 채팅 수업 도구

**기능 1(벤더 앱급 AI 웹 클라이언트) 위에 얹는 수업 운영 계층.** 실시간 채팅을 기반으로 **다양한 방식의 수업**을 운영하는 도구. 아래는 **활용 예시일 뿐**이며, 교사가 수업 방식을 자유롭게 구성한다. **채팅 도구만으로 성립한다.**

- **실시간 채팅·소통(핵심):** 교사↔학생 · 학급/모둠 · 1:1 · 익명 질문 · 공지·Q&A (사람 간 소통 — AI 대화(기능 1)과 **같은 입력창, 대화 종류만 다름**. 채팅도 코어 로깅·안전필터 경유. 익명 질문의 '익명'은 **교사·학급 표시 기준**이고 안전 목적 기록은 유지된다 — 이 범위를 UI에 사전 고지해 오인을 막는다)
- *(예)* **출제 → 제출 → 피드백:** 문제·과제 출제·전송 → 학생 풀이·제출 → 개별·전체 피드백
- *(예)* **분임별(모둠) 수업:** 모둠 채팅·과제·산출물 공유·토의
- **학생 산출물 시연(학생 → 교사):** 학생이 자기 산출물(AI 대화·글·코드·이미지)을 **교사에게 전송**(통합 입력 모델의 '대상=교사') → 교사가 **본인 동의 확인·익명 처리 후** 학급에 시연(교실 빔프로젝터). 동의·익명화·미리보기는 기능 1 '대화 공유·시연'과 **동일 흐름을 재사용**하고, **학생별 시연 횟수를 교사에게 표시**해 편중을 막는다. 바당2 밖 화면(외부 앱·문서)은 **이미지로 캡처·첨부**(기능 1 멀티모달)
- **수업 자료 배포(교사 → 학생):** 자료(PDF·슬라이드·링크)를 학생 화면으로 배포 — 학생이 직접 확대·TTS·스크롤(기능 6 접근성)

### 기능 3 — 학습 상태·루프 인프라 (초개별화 엔진)

특정 교수법을 구현한 기능이 아니라, **수업 루프를 돌리는 데 필요한 상태·오케스트레이션 인프라**다. 기능 1(AI 대화)가 '대화'를 맡는다면, 기능 3은 그 대화가 **학습으로 누적되게 하는 층**을 맡는다 — 어떤 수업 방식이든 이 인프라 위에서 자기 루프를 돌린다.

- **지식 추적(상태)**: 학생별 숙달 상태를 성취기준·스킬 단위로 갱신·보존 — 세션이 끝나도 남아 다음 수업의 출발점이 되는 **학습 상태 모델**(대화 히스토리와는 다른 것). *(단계 전략)* 시범 단계는 **형성평가 정오답 기반의 규칙형 숙달 판정**(도달/미도달·재도전 횟수)으로 시작하고, 축적된 응답 데이터로 **통계적 지식 추적 모델을 2단계에 고도화**한다 — 개통일부터 정밀 추적을 요구하지 않는다(**데이터가 모델보다 먼저다**)
- **스킬 맵(선행 자산)**: 지식 추적의 단위가 되는 **성취기준↔세부 스킬↔문항 매핑은 시스템이 아니라 교과 전문 작업**이다 — 시범 단계는 **1~2개 과목·단원으로 한정**해 에듀테크소프트랩(교육연구사)+교과연구회가 제작·검수하고, 확대 시 확장 마켓(기능 10)의 콘텐츠 축으로 개방한다. 국가 교육과정 성취기준 코드를 기본 골격으로 재사용 · **제작 마일스톤은 구축 일정과 동기화**(개발 중간 점검에 1개 단원분 → **시범 2단계(수업 맥락) 개시 전 시범 과목분 완성** — 콘텐츠 지연 = 기능 3 검수 불가)
- **루프 오케스트레이션(분기)**: 평가 결과에 따른 진행·분기(도달 → 다음 목표·심화 / 미도달 → 교정·재도전)를 정의·실행하는 워크플로 — 교수법 시나리오가 갈아 끼우는 규칙
- **교사 현황판**: 학급 전체 루프의 위치·정체 학생을 한 화면에 — 30개 대화창을 읽는 대신 집계로 개입 지점 결정
- **품질 이밸(상시·자동)**: 교과 문항·루브릭 기반 평가셋으로 튜터 응답 품질을 릴리스마다 자동 측정 — 월 주기 릴리스(기능 1)의 **회귀 방지 장치**(평가셋 제작은 스킬 맵과 같은 교과 콘텐츠 작업). **에이전트 오케스트레이션으로 팬아웃 실행**(§3-3)
- **문항 검증(출제 직후·자동)**: 만든 문항을 **다른 에이전트가 실제로 풀어보고 검증**한다 — *정답이 맞는가 · 복수 정답 소지가 있는가 · 난이도가 의도대로인가 · 오개념을 실제로 유도하는가.* **출제 오류를 학생에게 나가기 전에 잡는다**(기능 8 출제와 연계 · §3-3 오케스트레이션)
- **효과검증 데이터(학기 단위)**: 도달/미도달·성장 데이터의 정량 축적 — "잘 설계된 AI가 오른다"를 검증할 §5 효과검증(연구용역)의 근거(품질 이밸과 주체·방법·주기가 다른 별개 층위)
- **도구 인터페이스(MCP)**: 위 상태·분기를 튜터 에이전트가 쓰는 **도구**로 노출한다 — *숙달 상태 조회 · 스킬 맵 조회 · 형성평가 문항 요청 · 채점 결과 기록 · 다음 활동 분기 요청*. 교수법 시나리오는 **도구 조합과 시스템 프롬프트를 갈아끼워** 구성하므로 코어를 고치지 않는다(§3-3). 도구 반환값에도 데이터 최소화 원칙을 적용해 **가명 토큰과 상태 요약만** 돌려준다(§3-5)
- **연계**: 문항·루브릭은 기능 8(수업설계·평가), 활동 기록은 학습기록(기능 4)에 적재 — **학습기록에는 활동·평가 결과 이벤트(요약)만 담고** **대화 전문·중간 산출 원본은 싣지 않는다**(전문 로그에 참조 ID로만 연결, §3-5). 모든 상호작용은 코어 게이트웨이 경유(로깅·안전). **수업 중 루프에 서술형 채점 확정이 필요한 경우의 간이 채점 확정**(AI 1차 채점안 승인/수정 → 즉시 분기)**은 수업 화면에 중복 배치** — 유일한 기능 중복 예외(세션정책 §5-3, 전체 채점 워크벤치는 기능 8·업무 화면)

> **대표 시나리오 — 완전학습 루프:** 「수업 → 형성평가 → 도달: 다음 목표·심화 / 미도달: 오개념 진단 · 맞춤 개별 학습(기능 1이 실시간 생성 — **텍스트·수식·코드는 모델이, 그림은 「그림공방」이, 음성은 TTS가, 문서는 「문서공방」이 만든다.** 모델 출력 자체는 텍스트 전용 — 기능 1 「모달리티 사실」) → 맞춤 재출제」를 학생마다 실시간으로 — 학급 30명이면 30개 루프가 각자의 속도로. **교정의 목표(무엇을 보완할지)는 출제 설계에서 결정**되고(스킬 맵·오답 태깅), **교정의 콘텐츠(어떻게 가르칠지)는 그 목표를 입력으로 실시간 생성**된다 — 즉흥 판단이 아니라 사전 설계된 뼈대 위의 생성이다. 단 이는 **이 인프라 위에서 도는 활용 시나리오 중 하나**일 뿐이며, 토론 루프·프로젝트 단계 진행 등 다른 수업 방식도 같은 인프라로 구성할 수 있다 — 특정 교수법을 코어에 고정하지 않는다(§3-3 원칙).

### 기능 4 — AI 학습기록(누가기록) 관리

전체 송수신 로깅(코어, §3-3)을 **학생·교사가 원하는 형태로 가공·제공**. **학습활동 기록·분석은 바당2 대시보드에서 완결**(§3-2. **전문 원본은 교육청 로그에 별도 보관**).

- **학생용**: 나의 AI 학습일지(주제·기간별), 성장 리포트
- **교사용**: 학급/학생별 활용 분석, 자주 묻는 질문·오개념, 활동 요약 — 학생 기록 열람은 **단일 열람 게이트** 경유(집계·요약 기본 · 원문은 별도 스코프+사유+승인, 세션정책 §7-3)
- **누가기록**: 학생별 AI 사용 이력 누적(무엇을·어떻게 사고했는지)
- **생기부 반영 근거** 요약·발췌 — 단 **교사의 관찰·확인을 거친 참고자료**로만 제공(로그의 자동 전재·자동 기재 금지, 기재 주체는 교사), 생기부 활용 가능성은 학생·학부모에게 **사전 고지**, 반영 범위·기준은 제주교육AI플랫폼 운영위원회 거버넌스(§5)로 확정 / 기간·과목·주제 필터·템플릿·내보내기(PDF·표)

### 기능 5 — 안전·개인정보 보호

미성년 보호를 위한 공교육 필수 장치. 코어 안전필터(§3-3)·거버넌스(§3-5)를 *기능*으로 구현 — **안전·개인정보는 항상 ON**(교사가 끌 수 없음).

**안전·위기 감지**

- 연령별(초·중·고) **차등 콘텐츠 필터**
- **자해·학대·폭력·정신건강 위기 신호 탐지 → 담임·상담교사 알림** — 위기 신호는 **민감정보(건강, 개인정보보호법 §23) 처리**이므로 별도 법적 근거·고지 위에서 운영: ① 기계 탐지는 *선별*일 뿐 **후속 조치는 반드시 사람(상담교사)의 확인**을 거침 ② 알림 수신자·열람은 최소 범위 + 전건 감사로그 ③ **오탐 판정 시 해당 플래그는 학생 기록에서 삭제**(이의 절차 포함) ④ 알림 이후 개입 프로토콜(상담 연계·보호자 고지·신고의무 관계)은 위기감지 운영 지침으로 별도 정립 — PIA 필수 검토 항목
 - **다중 검증으로 오탐을 거른다:** **오탐이 많으면 상담교사가 지쳐 알림 자체를 무시하게 된다** — 위기감지의 가장 현실적인 실패 모드다. 탐지 신호를 **서로 다른 관점의 에이전트가 재검**(표현·맥락·이력)하고 **합의된 것만 사람에게 올린다.** ***"후속 조치는 반드시 사람"* 원칙은 그대로 유지하면서, 사람의 시간을 지킨다.** 건수가 적어 비용 부담도 작다. 재현율 하한·오탐 상한 수치는 §6.
- 시험부정·욕설·타인비방 경고, **프롬프트 인젝션·탈옥(jailbreak) 방어**
- **학업 진실성:** 산출물 진위는 **과정 증빙**(바당2 경유 산출물의 세션 토큰·산출물 표식 — §3-1)으로 구조적으로 입증 · AI 사용 투명성 표시. *(AI생성·표절 검사(카피킬러 연계)는 판별 도구의 오탐 신뢰도 문제로 도입하지 않음 — 2026.7.12. 삭제)*
- **딥페이크·생성물 오남용 방어:** 이미지 편집 모델은 **인물 일관성을 유지한 채 대화형으로 편집**하므로, 얼굴 사진 한 장이 곧 학교폭력 도구가 될 수 있다. ① **얼굴이 포함된 이미지는 편집 도구에 투입 불가 — 예외 없음**(입력 필터가 유일한 방어선 · 학생·업무 세션 공통) ② **생성·편집물 전량 로깅 + 출처 표식(워터마크)** ③ **연령별 유해성 필터를 프롬프트·결과 양방향에 적용** — 상세 §3-5 「새 위험 축 — 딥페이크」

**개인정보 보호·비식별화**

- 벤더 전송 전 **PII 자동 마스킹·비식별화** ("벤더엔 익명 토큰만"의 실제 구현) — **규칙(정규식)+경량 NER 기반 온프레미스 비식별 엔진**(오픈소스). **암호화·재식별은 KMS/HSM·매핑 룩업**으로 분리(생성형 AI 아님 — 상세 §3-5)
- **비전 비식별:** **이미지·영상 경로의 얼굴·이름표·손글씨 이름은 텍스트 비식별로 걸러지지 않는다.** 학생 이미지 입력 · pptx 내 삽입 사진 · 「그림공방」 · 「라이브튜터」 · 「업무자동화」 화면 캡처의 **다섯 경로**에 **얼굴 탐지·마스킹 + OCR 후 텍스트 재투입 + EXIF 제거**를 공통 적용한다 — 상세 §3-5, 인프라는 §3-4 ④(경량 비전 가속기 · **LLM 추론용 GPU 팜은 여전히 불필요**)
- **동의·법적 근거 관리**: 처리 근거 이원화 — 수업·안전 등 필수 처리는 공공기관 소관업무(법 §15①3호 등, 법률 자문으로 확정), 그 외(민감정보·연구·공유 활용)는 동의 기반. **동의 거부 학생에게도 불이익 없는 대체 학습 경로를 보장**(AI 접근 공적 보장의 전제). 보관기간 자동 만료·파기, 본인(및 법정대리인) 데이터 열람·삭제 요청
 - **동의 설계의 정밀화:**
 - **만 14세 미만**(초등 전원 + 중1 일부): §22-2①은 **법정대리인 동의 + 「동의했는지 확인(검증)」** 을 **이중으로** 요구한다 — **학생이 클릭하는 동의 UI로는 확인 요건을 충족하지 못한다.** 확인 수단(가정통신문·본인인증·NEIS 연계 등)과 **학기별 갱신 절차**를 운영 가능한 수준으로 설계해야 한다(§6).
 - **포괄·약관형 동의는 무효에 가깝다** — **목적별·항목별로 구분된 명시적 별도 동의**를 받는다(이루다 2021 · 비대면진료 의결의 §22① 위반 판단).
 - **국외이전 동의는 별도 동의다**(§28-8①1호) — 일반 수집·이용 동의와 **물리적으로 분리된 동의 도구**여야 하며, **5개 사항(이전 항목·국가·시기·방법 / 이전받는 자 / 이용목적·보유기간 / 거부 절차)을 사전 고지**해야 한다(§28-8②). **국내 리전 처리로 구성하면 이 부담 자체가 사라진다**(§6 최우선 확인).
 - **⚠️ 동의로 메울 수 없는 것 — 프롬프트 속 제3자 정보.** 학생이 친구 이름을, 교사가 다른 학생의 상담 내용을 쓰는 순간, **이용자 전원의 동의로도 그 데이터에는 아무 근거가 없다.** 이는 **동의가 아니라 입력 필터(비식별)·이용정책·교육으로만 통제된다**(§3-5 해제 불가 구간).
 - **⚠️ 동의는 안전조치 의무를 면제하지 못한다**(§23②→§29 · §28-8④) — *"동의를 받았으니 비식별은 안 해도 된다"* 는 성립하지 않는다.

### 기능 6 — 접근성·형평성 지원

**「모든 학생의 AI 접근 공적 보장」의 실질.**

- **음성은 '접근성 보조'가 아니라 '실시간 대화급'이다.** 기능 1의 「음성대화」·「듣는요약」·「라이브튜터」가 **개통 필수**이므로 **자체 스트리밍 음성 스택이 코어 요건**이다. 실수요는 접근성에 그치지 않는다 — **영어 회화 연습 · 저학년 · 특수교육 · 다문화 학생 모국어 대화**.
 - **음성대화**: 자체 **스트리밍 STT → 텍스트(비식별·안전필터 경유) → TTS**. **벤더 직결형 실시간 음성 API는 게이트웨이를 우회하므로 사용하지 않는다**(§3-1). *"벤더가 안 주니 저품질로 참는다"가 아니라 "우리 스택으로 동급 지연을 낸다"* — **지연 목표치는 §6·ISMP에서 수치로 확정**한다.
 - **듣는요약**: 학습 자료·수업 내용을 **2인 대담 음성**으로 생성(대본 생성 + 2인 TTS). 추가 인프라가 거의 없으면서 **읽기에 어려움을 겪는 학생·시각장애 학생·통학길 복습·학부모 소식 요약**에 바로 쓰인다 — **투입 대비 산출이 가장 좋은 접근성 기능**이다.
 - **라이브튜터**: 카메라로 **손글씨 풀이·실험 장면을 비추며** 대화 — 완전학습 루프(기능 3)의 **오개념 진단이 실시간 시각으로** 이뤄진다. **비전 비식별(§3-5) 경유가 전제**다.
- 자막·큰글씨 (장애학생)
- 다국어·모국어 보조 (다문화학생), 저사양 기기·저대역 환경 대응 — **저사양 단말은 「바로분석」(브라우저 실행)에서 「실행실」(서버 실행)로 자동 폴백**한다(기능 1)
- **웹 접근성 법정 준수:** KWCAG(한국형 웹 콘텐츠 접근성 지침) 준수와 **웹 접근성 인증 취득**을 RFP 필수 산출물로 명시 — 「모든 학생의 AI 접근 공적 보장」의 법적 형식(장애인차별금지법)

### 기능 7 — AI 토큰 관리

코어 모델 라우팅(§3-3)의 **비용·정책 통제 심화**.

- **벤더별**(Claude·OpenAI·국산 등) 사용량·비용 집계
- **사용자별**(학생·교사, 학교) 토큰 cap·잔량·소진 관리
- 단위별 한도: 학급 / 기간(일·월·학기)
- 실시간 사용량·비용 대시보드, 이상 사용 탐지, 학교·벤더별 정산 근거

> **🟦 회계 구분 — API 사용료는 구축비가 아니다.** **① 구축비(FP 산정 · 2027 본예산 · SW 개발)** 와 **② AI API 사용료(벤더와 별도 계약 · 종량·선불 · 2028 본예산 시범 운영비)** 는 **섞이지 않는다.** 산정이 필요한 이유는 **「연간 API 계약 규모를 정확히 잡기 위해서」** 이지 비용 위험을 경고하기 위해서가 아니다 — **정당한 사용료는 당연히 지불한다.**

**cap 정책 — 「사전 요청 · 교사 수용 → 차단」*

> **cap의 목적은 「폭주 방지」와 「이상 사용 탐지」이지 「절약」이 아니다.** 정당한 사용료 지불이 전제이므로 **cap은 넉넉하게 잡는다.**
>
> **핵심 설계 — 「예고 없이 끊지 않는다」.** 「차단 금지·경량 모델 폴백」이라는 대안의 진짜 목적은 *"차단 → 벤더 앱 우회 → 학생 실명이 비식별 없이 유출"* 이라는 **최악 경로를 막는 것**이었다. 그런데 **무한 폴백은 cap을 무력화**하고(무제한 사용과 같다) 폭주도 못 막는다. **폴백은 품질을 떨어뜨려 버티게 하는 것 — 학생은 결국 좋은 것을 쓰러 밖으로 나간다.**
> **➡️ 수단을 바꾼다: 「소진 전에 미리 요청하고, 교사가 제때 채워준다.」** 예고 없는 단절이 없으므로 우회 유인이 사라지고, **cap은 실제로 작동한다.** ㉮의 정신을 폴백보다 더 잘 구현하는 방식이다.

| 소진율 | 조치 | 수용 주체 |
| ---: | --- | --- |
| **80%** | 알림 | — |
| **95%** | 경고 + 잔량 표시 + **[추가 배정 요청] 버튼 활성화** | — |
| **요청 시** | 학생 → **교사가 수용**(수업 중 즉시 처리) · 교직원 → 관리자 수용 | 교사 / 관리자 |
| **100%** | **차단** *(요청이 없었거나 거부된 경우)* | — |
| **비정상 폭주** | **즉시 중단 — 경고·요청 경로 없음**(아래) | — |

- **교사가 수용 주체인 것은 교육적 판단이 필요하기 때문이다** — *이 학생이 정말 더 필요한가, 낭비하고 있는가.* 관리자 결재로 올리면 수업 중 처리가 불가능하다.
- **자율 맥락(가정·심야)은 비동기 승인이 성립하지 않는다** — 밤 10시에 승인을 기다리게 하면 그 학생은 벤더 앱으로 간다. **자동 수용 규칙**(1일 1회·일정량까지 자동 배정, 그 이상은 담임·학교 관리자 승인)을 기본값으로 둔다. 수치는 ISMP·운영위원회에서 확정.
- **관리자 콘솔(기능 11)에서 요청·수용 이력을 조회**하고, 반복 요청자는 **이상 사용 탐지**로 넘긴다.

**폭주(runaway) 방지 — cap 초과와 성격이 다르다**

> **핵심 구분:** **정상 사용의 cap 초과는 「사용자가 원한 사용」**이므로 경고 후 차단하고 즉시 복구 경로를 연다. **비정상 폭주는 「아무도 원하지 않은 호출」**이므로 **경고 없이 즉시 중단**한다. 이 둘을 섞으면 폭주를 못 막거나 정상 사용까지 끊는다.

| 폭주 유형 | 방어 |
| --- | --- |
| **무한 루프**(도구 호출 → 해석 실패 → 재호출 반복) | **최대 반복 횟수 · 타임아웃** → 즉시 중단 |
| **재귀 팬아웃**(에이전트가 또 에이전트를 만든다) | **팬아웃 깊이 1단계 제한** — 하위 에이전트는 다시 팬아웃할 수 없다 |
| **오류 재시도 루프** | 재시도 상한 · 지수 백오프 · **서킷 브레이커** |
| **확장이 유발하는 폭주** | 확장별 **호출 상한·예산 스코프** — 해당 확장만 중단(기능 10·11) |
| **반복 요청 남용** | 레이트 리밋 · 사용자별 동시 작업 수 제한 |

- **전체 안전망:** 기관 단위 **일 소진율 감시 → 알림 → 자동 스로틀 → 킬 스위치**(기능 11 관리자 콘솔).

**에이전트 팬아웃의 비용 통제 (§3-3 오케스트레이션 대응)**

> **cap 산정의 전제가 바뀐다.** 「사용자별 토큰 cap」은 **요청 1건 = API 호출 1회**를 암묵 전제로 했으나, 에이전트 오케스트레이션에서는 **요청 1건이 호출 수백 회**가 될 수 있다(에이전트 1명 ≠ 호출 1회 — 각자 여러 턴을 돈다). 이는 **위험이 아니라 산정 전제의 문제**다 — 배수를 알아야 계약 규모를 잡을 수 있다.

- **작업 예산(task budget)** — 팬아웃 작업은 **시작 전에 토큰 상한을 정하고, 초과하면 중단**한다. 사용자 cap과 **별개의 2차 방어선**이다.
- **팬아웃 폭 제한** — 기본값을 정책으로 고정한다(예: 교차 채점 3인 · 위기감지 검증 3인). 그 이상은 **관리자 승인**(기능 11).
- **명시적 호출만** — 팬아웃 기능은 **사용자가 버튼을 눌러 시작**하며, **예상 비용을 사전에 표시**한다. **자동·백그라운드 팬아웃은 두지 않는다**(아무도 모르게 소진된다).
- **작업 단위 정산** — **상관 ID(작업 ID)** 로 N개 호출을 하나의 작업으로 묶어 집계한다(§3-3). 묶지 않으면 *"이 채점에 얼마 들었나"* 를 답할 수 없다.
- **원칙 — 값어치가 확실한 곳에만.** ✅ 서술형 교차 채점(기능 8 — 교사 시간이 훨씬 비싸다) · 위기감지 오탐 검증(기능 5 — 건수가 적다) · 문항 검증·품질 이밸(기능 3 — 출제 시 1회) · 깊이조사(기능 1 — 명시적 요청 시). ❌ 일반 대화 · 자동 백그라운드 작업.
- **「학생 1인당 API 사용료 산출」은 팬아웃 배수를 반영해 재산정**한다 — **연간 벤더 계약 규모의 근거**가 된다(§6).

### 기능 8 — 교사 수업설계·평가 지원

교사가 **수업을 준비(설계)** 하고 **학생 결과물을 평가**하는 일을 AI가 돕는, **교사 전용 작업대(workbench)**. 학생 대화 화면과 분리된 *교사 전용 작업 화면*이며(둘 다 기능 1 인터페이스 위에서 동작), 만든 결과물은 그대로 수업에 투입(기능 1·2)하거나 동료와 공유(기능 9)한다.

> **바당엔 없던 'AI 에이전트'가 바당2엔 항상 대기한다:** 교사가 목표만 주면 에이전트가 **본 작업(수업 설계·자료·문항·채점)을 직접 수행해 결과를 만들고, 교사는 고르고 다듬어 결정**한다.

**① 수업 설계(준비)**

| 교사가 하려는 일 | AI에게 받는 도움 (예) |
| --- | --- |
| 내일 '광합성' 차시를 준비한다 | **성취기준**을 넣으면 → 학습목표·40분 수업 흐름·핵심 발문 5개 **초안** |
| 설명자료·활동지가 필요하다 | 주제·분량 지정 → 설명글·예시·읽기자료·활동지 생성 |
| 우리 반에 느린 학습자·빠른 학습자가 섞여 있다 | 같은 자료를 **쉬운 버전 / 심화 버전**으로 **수준별 변형** |
| 시험·과제를 출제한다 | 유형(객관식·서술형·**수행과제**)·난이도·문항 수 지정 → 문항+정답+해설 |
| 채점 기준이 필요하다 | 성취기준 기반 **루브릭(척도·배점)** 자동 작성 |
| 이 차시에 AI 튜터를 쓰고 싶다 | 튜터 모드 **시스템 프롬프트**(소크라테스식·토론·교과 맞춤) 초안 (기능 1 연계) |

**② 평가(채점·피드백)**

| 교사가 하려는 일 | AI에게 받는 도움 (예) |
| --- | --- |
| 서술형 답안 30개를 채점한다 | 교사가 만든 **루브릭 기준**으로 **1차 채점안+근거** 제시 → 교사가 확정 · **교차 채점**(아래) |
| 한 학생에게 피드백을 쓴다 | 강점·보완점 짚은 **맞춤 피드백 문구** + **오개념 진단** |
| 시험 결과를 분석한다 | 학급 전체 응답에서 **공통 오류·성취 분포** 요약 → 다음 수업으로 환류 (기능 4 연계) |

> **교차 채점:** *"AI가 1차 채점안을 제시"* 는 **AI 한 명의 판단**이다. **답안 하나를 서로 다른 에이전트 3명이 독립 채점**하고, **판정이 갈린 답안만 교사에게 올린다.**
> - **이는 실제 평가의 「복수 채점자」 방식을 그대로 옮긴 것**이며, **교사의 검토 부담을 극적으로 줄인다** — 30개 답안 중 의견이 갈린 소수만 보면 된다. 채점 신뢰도도 함께 오른다.
> - **채점 확정 권한은 여전히 교사에게 있다**(기능 3 거버넌스 「자동화된 결정 방지」) — 교차 채점은 *교사가 어디를 봐야 하는지* 를 알려줄 뿐 결정을 대신하지 않는다.
> - **비용:** 답안 1개당 3회 호출. **작업 예산(§3-3)** 안에서 실행하고, 팬아웃 폭(기본 3인)은 관리자 콘솔(기능 11)에서 조정한다.

- **프롬프트 템플릿 라이브러리:** 과목·학년별 수업 설계·평가용 프롬프트 모음 — 교사는 위 작업을 **빈칸 채우듯** 시작

**③ 축적된 수업 자산의 재활용*

| 교사가 하려는 일 | AI에게 받는 도움 |
| --- | --- |
| **기존 공문·자료**를 활용한다 | **hwpx·hwp(레거시) 읽기** → 요약·분석·재작성 |
| **기존 pptx 수업자료**를 활용한다 | **pptx 읽기** → 활동지로 변환 · 쉬운/심화 버전 분리 · 형성평가 문항 생성 |
| 산출물을 **공문서 표준으로** 받는다 | **hwpx 생성** — 교육청 표준 공문·가정통신문 서식 적용 |
| 받은 초안을 **바로 고친다** | **「한글 편집기」** — 브라우저에서 hwpx를 서식 그대로 편집 → 저장 → 결재 |

> **왜 중요한가:** 벤더 앱은 hwpx를 **읽지도, 만들지도, 편집하지도 못한다.** AI가 초안을 줘도 결국 한글 프로그램에 옮겨 적어야 한다면 **업무는 절반만 자동화된 것**이다. **읽기·쓰기·편집이 브라우저에서 완결**되어야(기능 1 「문서공방」·「한글 편집기」) 축적된 자산이 살아나고 §0의 「벤더 구독 재가입 최소화」가 성립한다. **공문서 법정 표준이 hwpx로 확정**(2026.10. 의무화)되었으므로 이는 정책 준수이기도 하다. *(단, 공문·자료에는 학생 명부·성적이 담긴 경우가 흔하다 — 파싱된 텍스트는 **반드시 비식별 엔진을 통과**시킨다. 코어의 문서 파서가 그 검문소다.)*

> **교직원 업무 활용(같은 기본 AI로):** 위 작업대(workbench)와 **동일한 기본 AI**(기능 1)로 교사·교직원의 **일반 행정 업무**(공문·가정통신문 초안·자료 요약·데이터 분석 등)도 처리한다. **업무 특화 기능은 확장팩**(기능 10)으로 장착. 이 활용은 **업무 세션·업무 화면 소속**이다(§3-6) — 학생 세션과 정책 프로파일이 분리되고(위기감지 대신 고위험 업무 유형 통제·MFA·대량 조회 제한·열람/갱신 전건 감사), 교직원 로그 거버넌스는 세션정책 §9를 따른다. 외부 ChatGPT·Claude에 흩어 쓰던 업무를 **하나의 안전한 공공 관문으로 통합**한다.
> **업무 세션 전용 기능(기능 1):** **「AI 선택」 자유** · **「예약작업」**(정기 리포트·가정통신문 초안 자동화) · **「업무자동화」**(행정 시스템 화면 조작 — 단 **정공법은 정식 API 연동(내부 MCP)이고 화면 조작은 폴백**이며, **화면 캡처가 곧 학생 개인정보 화면**이므로 비전 비식별과 반드시 함께 개발한다). **학생 세션에는 이 셋이 노출되지 않는다.**

### 기능 9 — 수업 나눔 (교사 협업·공유)

교사가 자신의 수업을 공유하면 다른 교사가 가져다 **자기 수업에 바로 활용**한다 — 교사 채택·확산의 핵심 동력.

> **공유 대상 = 교사 저작물뿐(학생 데이터 미포함).** 프롬프트·문항·루브릭·자료·수업 흐름 템플릿 등 **교사가 만든 것**만 공유한다 — **학생 답안·대화·산출물은 어떤 형태로도 포함하지 않는다.** 학생 산출물을 남에게 보이는 일은 **교실 내 시연**(기능 1 '대화 공유·시연' · 기능 2 '학생 산출물 시연')에서만 일어나며, 그쪽의 **건별 동의·익명 기본값**으로 다룬다.

- **수업 패키지 공유:** 수업 흐름 **템플릿**(대화 로그가 아님)·출제·모둠 구성·AI 모드·프롬프트·자료·루브릭을 **수업 단위로 묶어** 공개/공유
- **시스템 프롬프트 공유:** 교사가 만든 수업용 시스템 프롬프트(튜터·토론·교과 맞춤 등)를 **단독으로도** 공유 → 다른 교사가 그대로/수정해 적용 (제작은 기능 8 '튜터 모드 시스템 프롬프트', 적용은 기능 1과 연계)
- **가져오기·복제·수정:** 다른 교사가 클릭으로 가져와 자기 학급에 맞게 고쳐 사용
- **탐색:** 과목·학년·단원·**성취기준**별 검색, 추천·인기
- **검수·안전:** 학생 데이터가 애초에 포함되지 않으므로 **검수는 실수 대비**다 — 게시 전 **개인정보 자동 스캔**(기능 5)으로 교사가 자료에 남긴 학생 이름 등을 걸러내고, 검출 시 게시를 보류한다. 우수 수업 큐레이션·아카이브 등재 시 재확인.
- (확장) 교육청 **우수 수업 아카이브** — 제주형 수업 자산 축적

### 기능 10 — 서드파티 확장 모듈(콘텐츠) 마켓 운영

> 바당2의 **지속 발전**은 서드파티 확장이 끊임없이 나오는 **생태계**에 달렸다 — **코어 기능의 더 나은 대안까지** 확장으로 받아들이며(§3-3), 채택 선택은 학교·교사가 한다. 표준·개발환경·마켓으로 그 환경을 조성한다.

#### 🟦 「나만의 AI」와 3단 사다리 — 교사가 닿을 수 있는 0단

> **문제:** **교사가 자기 AI를 만들어 학생에게 배포하는 경로**가 필요하다. 그렇지 않으면 남는 것은 「말투설정」(문체만) · 기능 8(시스템 프롬프트 *초안* 생성) · 기능 10(**2단계 심사** 후 마켓 게시)뿐이다. 즉 교사가 *"우리 반 분수 튜터"* 를 만들려면 **심사를 받아야 했고 — 그러면 아무도 만들지 않는다.** 「확장팩」·「수업 나눔」·「확장 마켓」이 전부 **교사가 닿을 수 없는 상층부**만 남는다.

**「나만의 AI」는 두 유형이다 — 대화가 필요한 일과, 문서만 필요한 일은 다르다**

> **대화형 챗봇 하나로는 부족하다.** 반복 업무(가정통신문·계획서·보고서)에서 **교사는 대화하고 싶은 것이 아니라 빈칸만 채우고 문서를 받고 싶다.** 챗봇형보다 **도구형의 실무 효용이 더 클 수 있다.**

| 구성 | 내용 |
| --- | --- |
| 이름·썸네일 | *"3-2반 분수 튜터"* · 썸네일 **AI 생성** |
| 역할 지시 | 시스템 프롬프트 — **기능 8이 초안을 만들어 준다** |
| 지식 | 「지식창고」에서 자료 선택 · **출처 공개 여부** 설정 |
| 도구 | 실행실·바당검색·그림공방 등 켜고 끄기 |
| 시작 질문 | 버튼으로 노출(최대 4개) — 무엇을 물을지 막막한 학생·교사를 위해 |
| 미리보기 | **만들면서 바로 테스트** (대화 미저장) |
| 배포 | **자기 학급에 LTI로 배치 — 심사 없음** |

**② 도구형 — 폼을 채우면 문서가 나오는 AI** 

| 단계 | 내용 |
| :---: | --- |
| **1** | **양식 등록**(학교 hwpx 서식) + 도구 이름·설명 |
| **2** | **입력항목 설계** — 텍스트형·선택형 · 라벨·설명·예시·필수 여부 |
| **3** | **출력항목 → 양식 매핑** — `{제목}`·`{인사말}` 위치 지정 · 서식 조정 |
| **4** | **프롬프트 자동 생성**(AI가 1~3단계를 분석해 작성) → **테스트** → 게시 |

> **결과:** 교사는 **빈칸만 채우고 [생성]** 을 누른다 → 학교 공식 서식에 맞춘 **hwpx 문서**가 나온다 → 「한글 편집기」에서 부분 수정 → 「문서함」에 저장 → 결재. **대화 없이 업무가 끝난다.**

**3단 사다리 — 심사는 위로 갈수록만 붙는다**

| 단 | 무엇 | 범위 | 심사 |
| :---: | --- | --- | :---: |
| **0** | **나만의 AI** | 자기 학급 | **없음** |
| **1** | **수업 나눔**(기능 9) | 교사 간 공유 | **없음** *(개인정보 자동 스캔만)* |
| **2** | **확장 마켓**(기능 10) | 전체 배포·상용화 | **2단계 심사** |

> **심사 없이도 안전한 이유:** 교사가 무엇을 조합하든 **코어 안전필터·비식별·전량 로깅을 우회할 수 없다**(§3-3 불변 원칙). **심사는 *코드*를 볼 때 필요한 것이지 *프롬프트*를 볼 때 필요한 것이 아니다** — 시범 단계가 허용하는 **선언형 확장**(코어 내 임의 코드 실행·PII 접근 없음)의 정의에 「나만의 AI」가 정확히 들어맞는다.
> **효과:** 0단이 생기면서 *"기능 9에서 현장 검증을 거친 자산이 기능 10으로 **승격**한다"* 는 기존 설계가 비로소 **바닥부터 이어진다** — 교사가 만들고 → 동료가 쓰고 → 좋은 것이 마켓으로 올라간다. **「개통 기본 스킬 세트」(아래)는 이 사다리의 종잣돈**이다.

> **기능 9(수업 나눔)와의 경계 — 「수업 자산」 vs 「상용화 수준 패키지」.** 교사가 만든 것이라도 둘은 다르다. **기능 9**는 교사가 자기 수업에 쓰던 **수업 자산**(프롬프트·문항·루브릭·자료·수업 흐름 템플릿)을 **심사 없이 교사 간 그대로 공유**하는 것이고, **기능 10**은 **상용화 수준으로 패키징된 앱·콘텐츠·스킬**(설치·설정·버전·지원이 따르고 LTI·MCP로 코어에 결합)을 **2단계 심사를 거쳐 마켓에 게시**하는 것이다. 즉 가르는 기준은 **제작 주체(교사/업체)가 아니라 완성도와 결합 방식**이다. 기능 9에서 현장 검증을 거친 자산이 다듬어져 기능 10으로 **승격**하는 경로를 둔다.

- **확장 패키지 = 유통 단위:** 확장은 도구 하나가 아니라 **한 덩어리로 심사·배포·설치되는 패키지**다. ⓐ **매니페스트**(이름·버전·작성자·요구 권한 스코프·호출할 도구 선언) ⓑ **시스템 프롬프트**(수업 역할) ⓒ **MCP 도구·데이터 연결** ⓓ 화면을 갖는 경우 **LTI 도구 등록** ⓔ 콘텐츠·문항·루브릭을 담는다. *(코어 내부 확장점인 **어댑터**와는 층위가 다르다 — §3-3.)*
- **패키지 규격은 「발명」이 아니라 「표준 조립」이다:** 위 다섯 층 중 **ⓐ 매니페스트만 자체 정의**하고 나머지는 국제 표준을 그대로 쓴다 — ⓒ **MCP** · ⓓ **LTI 1.3** · **ⓔ 문항·루브릭 = QTI 3.0 프로파일** · **성취기준 = CASE** · 자료·메타 = **Common Cartridge·LOM**(1EdTech). **문항을 제주 고유 포맷으로 만들면 에듀테크 업체는 오지 않고, AIDT·타 시도 콘텐츠도 수용·수출할 수 없다** — 「제주 에듀테크 산업 육성」은 표준 위에서만 성립한다. 상세 근거·벤더 형식 비종속은 **§3-3** 참조.
- **붙는 방식:** 사람이 쓰는 화면은 **LTI**, 모델이 쓰는 도구·데이터는 **MCP** — 아키텍처는 §3-3.
- **심사가 무엇을 보는가:** ① 계획서 심사 = **매니페스트 선언**(요구 권한·호출 도구가 목적에 비해 과하지 않은가) · ② 최종 콘텐츠 심사 = **선언과 실제 동작의 일치**(선언하지 않은 도구 호출·외부 전송이 없는가). **버전이 바뀌면 재심사**하고, 심사 통과본과 배포본의 버전 동일성을 마켓이 보증한다.
- **이름 충돌 방지:** 확장 패키지는 **네임스페이스**(개발자/패키지명)를 갖는다 — 서로 다른 개발자가 같은 이름의 도구를 등록해도 충돌하지 않는다.
- *(예시)* **과목 AI 튜터**(수학 단계풀이·영어 회화) · **평가 도구**(서술형 자동채점·루브릭) · **특수교육·다국어 보조** · **제주 4.3·생태 학습**(지역 특화) · **교사 생산성**(수업안·가정통신문 생성)
- **개발 환경:** 공개 API·SDK·문서 → 에듀테크 기업·교사 개발자가 쉽게 제작·등록 (**서드파티 확장 모듈 제작 방법·가이드는 추후 GitHub 공개**)
- **확장 마켓:** 2단계 심사(① 계획서 ② 최종 콘텐츠)·배포·발견·평점 → 좋은 콘텐츠 확산·**제주 에듀테크 산업 육성**
- **탐색 축은 둘이다 — 수업 축과 업무 축.** 수업 축(과목·학년·단원·성취기준)만으로는 **교직원 업무**를 찾을 수 없다. **업무 축**을 함께 둔다: **월(학사 일정 기반 추천)** · **부서** · **업무 분야** · **학교급**.
 - **「월별 추천」이 특히 실효적이다** — *3월엔 학급운영계획, 5월엔 체험학습 가정통신문, 12월엔 학년말 업무*. 교직원 업무는 **학사 일정에 따라 반복**되므로, 필요한 때 필요한 도구가 앞에 놓인다.
- 기능을 **모듈로 추가·삭제**(교사 선택) — 학급·과목 맞춤 구성

#### 🟦 개통 기본 스킬 세트 — **빈 마켓 문제의 해법**

> **문제:** 개통일에 「확장팩」 메뉴만 있고 **장착할 것이 없으면** 기능 10은 껍데기이고, §0의 *"특화 기능은 확장팩으로 골라 장착한다"* 는 빈말이 된다. **마켓은 처음부터 진열되어 있어야 한다.**
>
> **해법 — 기본 스킬은 코어가 아니라 「교육청 제작 1st-party 확장팩」이다.** 코어에 박지 않는다. 이유는 §3-3이 이미 말했다 — *"코어 기능은 기본 보장선일 뿐 **만들어진 순간부터 낡는다**"*. **확장팩으로 만들어야 더 나은 민간 확장이 나왔을 때 학교가 갈아 끼울 수 있다.** 아울러 **교육청이 첫 번째 확장 개발자가 되어** SDK·규격을 자기가 먼저 써서 검증한다(§3-3).

**제작·발주 — SW 구축과 분리한다**

> **성격·주체·대가기준이 모두 다르다.** ① SW 구축은 **기능점수(FP)** 기반이나 콘텐츠 제작은 FP로 산정되지 않는다(**용역 대가** 체계) ② 스킬 설계는 *"시스템이 아니라 **교과 전문 작업**"*(기능 3 스킬 맵과 동일 논리)이라 SI 업체 역량이 아니다 ③ 무엇보다 **콘텐츠가 먼저 나와야 SW 검수가 된다**(*"콘텐츠 지연 = 기능 3 검수 불가"*).
>
> **➡️ 「교육 콘텐츠 제작」을 별도 트랙으로 발주한다** — 스킬 맵 · 평가셋 · **기본 스킬(위 3군)** · 서식·템플릿을 묶은 단일 트랙. **SW 구축과 병렬**로 진행해 콘텐츠 병목을 해소한다. **설계·검수는 교육청**(에듀테크소프트랩 교육연구사 + 교과연구회), **제작 실행은 용역**으로 역할을 나눈다 — 전부 자체 제작하면 인력 병목(스킬 맵·평가셋·기본 스킬이 **같은 사람들**에게 몰린다)에, 전부 용역이면 현장성 결손에 걸린다.
>
> **⚠️ 분리의 전제 — 「규격 조기 산출」:** 콘텐츠 제작은 **패키지 규격**(매니페스트 · MCP 도구 계약 · QTI/CASE 프로파일 · SDK)이 나와야 착수할 수 있다. **규격을 구축 계약의 조기 산출물(개발 착수 +1개월)로 못 박고, 중간 점검 게이트의 통과 조건**으로 둔다. 위 「표준 조립」 원칙 덕분에 구축사는 규격을 *창작*할 필요가 없고 **표준을 고르고 프로파일만 정의**하면 되므로 리스크가 작다.
>
> **계약 실무 3종:** ⓐ **저작권 교육청 귀속 + 2차 저작·수정·재배포 권리 명시**(마켓 게시·교사 수정이 전제 — 빠뜨리면 나중에 마켓에 못 올린다) ⓑ 교원 참여 비중이 크므로 **용역 + 교원 원고료·검수 수당의 혼합 구조** 검토(계약 부서 사전 협의) ⓒ **사업비 별도 계상** — FP에 잡히지 않으므로 「콘텐츠 제작비」 항목이 없으면 **통째로 누락**된다(§6).

> 개발자(업체·교사)가 만든 확장은 제주교육AI플랫폼 운영위원회(가칭)의 **① 계획서 심사 → ② 최종 콘텐츠 심사**를 통과해야 마켓에 게시된다. 심사에서 반려되면 보완 후 재신청한다 — 심사를 거치지 않은 확장은 학교에 배포되지 않는다.

> **단계적 도입·보안:** **외부 코드 실행형 확장은 전면시행 단계**(§5 5단계 심의 이후)에 적용한다. **시범 단계는 무료·교육청 직접 큐레이션**과 **선언형 확장**(LTI/MCP로 데이터·도구만 연결, **코어 내 임의 코드 실행·PII 접근 없음**)만 허용한다 — **「교육청 직접 큐레이션」의 실체가 곧 위 「개통 기본 스킬 세트」다.** 기본 스킬은 전부 **선언형**(시스템 프롬프트 + MCP 도구 연결 + 표준 콘텐츠)으로 성립하므로 이 제약과 충돌하지 않는다. 코드 실행형 확장은 **격리 런타임(샌드박스)·PII 비접근·코드 보안 심사**를 RFP 필수 요건으로 둔다(미성년 데이터 보호). 선언형(MCP) 확장도 안전과 동치가 아니다 — **도구 반환 데이터 경유 프롬프트 인젝션·데이터 유출 벡터**로 취급해, 도구 호출·반환 전문도 코어 로깅·안전필터를 통과시키고 확장별 **권한 스코프 최소화**·외부 전송 도구 사전 승인·이상 호출 탐지를 심사 기준에 포함한다.

> **격리 런타임은 「코어 공용 샌드박스」다.** 확장 전용이 아니다 — 기능 1의 「실행실」(학생·교사 코드 실행)·「문서공방」(파일 생성)이 **같은 런타임을 함께 쓴다.** 즉 **샌드박스는 확장을 위한 부대설비가 아니라 코어 인프라**(§3-4 부하 축 ⑤)다.
> - **공유:** 학생·교사 코드 실행 · 파일 생성(.docx·.xlsx·.pptx·.hwpx) · 서드파티 코드 실행형 확장 — **하나의 격리 런타임**을 쓴다.
> - **요건:** 컨테이너 격리(경량 VM급) · **네트워크 차단** · 시간·메모리 상한 · 파일시스템 격리 · 실행 전문 로깅.
> - **효과:** 확장 마켓을 위해 어차피 지어야 할 설비를 **코어 기능이 함께 쓰므로 중복 투자가 없다.** 반대로 이 샌드박스가 없으면 **「실행실」·「문서공방」·코드 실행형 확장이 모두 성립하지 않는다.**

| **확장 감독** | 전 사용자가 만든 **「나만의 AI」·도구 조회·미리보기·이용 횟수·공개 상태** — 문제 확장의 **즉시 비노출** |

| **운영** | 공지·알림 · 릴리스 노트 · 1:1 문의 접수 |

> **🔵 PII 처리 방식 — 「기본 On + 계층적 봉쇄 + 조건부 해제 + 전량 감사」*
>
> 상용 서비스 중에는 관리자가 **마스킹을 끌 수 있게(Off)** 한 것이 있다. 바당2의 결론은 **「전면 금지」도 「관리자 재량」도 아니다.**
>
> | 관리자가 고를 수 있는 것 | |
> | --- | --- |
> | **Mask** (가명 토큰으로 전송) · **Mask + 재식별**(출력단 실명 복원) · **Block**(민감정보 포함 시 차단) | ✅ 관리자 콘솔에서 선택 |
> | **해제(Off)** | ⚠️ **전역 토글 없음.** 목적·기간·항목·대상을 특정한 **요청 단위**로만 · **CPO 승인** · **TTL 만료** · **전량 감사** — §3-5의 6개 조건을 **전부** 충족할 때만 |
> | **해제 불가 구간** | 🔒 **만 14세 미만 · 민감정보(상담·정신건강·위기신호) · 고유식별정보 · 프롬프트 속 제3자 정보** — **어떤 승인으로도 열 수 없다** |
>
> **왜 「전면 금지」가 아닌가:** 조문상 마스킹은 의무형이 아니다(암호화만 의무형). *"마스킹은 법정 의무"* 라고 쓰면 **과장이고 역풍을 맞는다.**
> **왜 「관리자 재량」이 아닌가:** **동의는 안전조치 의무를 면제하지 못하며**(§23②→§29 · §28-8④), **프롬프트 속 제3자 정보는 이용자 전원 동의로도 원리적으로 메울 수 없다.** 해제는 **PIA에서 정당화 부담을 지는 설계 결정**이다.
> **➡️ 설계 목표는 「해제가 필요 없게 만드는 것」이다** — 「Mask + 재식별」이면 **교사는 실명 문서를 얻고 벤더는 실명을 보지 않는다**(§3-5).
>
> **권한 계층:** 교육청 관리자 / 학교 관리자 / 일반 사용자. **학교 관리자는 자기 학교 범위**에서만 조회·제어한다. 관리자의 모든 제어·열람 행위는 **전건 감사로그**를 남긴다(세션정책 §7-3 단일 열람 게이트). **PII 해제 승인은 학교 관리자 권한 밖이며 CPO 전속 사항이다.**

## 5. 추진 일정(안)

> **요약 일정** — 상세 일정·게이트·역산은 **「개발일정(종합)」(내부 문서)**, 추진 체계·단계별 업무는 **「개발TF 업무계획(안)」(내부 문서)** 참조. 핵심: **개발TF 2026.7.15 전 조직 · 기획+ISP·ISMP는 개발TF가 자체 수행해 2026-09-30까지 완료(산출 사업비 → 2027 본예산 반영) · 시범 운영 2028.1~5(1단계 업무·자율 → 2단계 수업 맥락) · 공식 개통 2028-09-01(2028학년도 2학기) · 제주교육AI플랫폼 운영위원회는 시범 개시 시 출범**. 단계 소요는 T0(예산 성립, 2027-01-01) 기준 상대 개월(T+n). 시범 단계 서술의 세션·맥락 용어(학생 세션·업무 세션·수업/자율 맥락)는 「세션정책」(내부 문서) §3 정의를 따른다.

**제주교육AI플랫폼 운영위원회(가칭):** 교원·학부모·외부 전문가(AI·교육·법률)·교육청으로 구성. **운영 거버넌스**(안전·윤리·개인정보(PIA)·생기부 반영·모델/콘텐츠 선정·효과 평가)를 자문·심의 — **시범 개시(서비스 개시, 2028-01-02) 시 출범**(개발 단계 의사결정은 개발TF 주관). 자율 맥락 실사용(위기감지·동의)이 시작되는 시점부터 거버넌스 공백이 없어야 한다.

## 6. 확정 전 확인 사항

- **바당 연계:** 구성도상 개방형 표준 연계(**SAML · OneRoster · LTI · xAPI · API Gateway**)가 이미 존재 → 바당2는 표준 도구로 결합 가능. 다만 **실제 LTI/API 개방 범위·권한 부여는 NIA(운영주체)와 협의**로 확정 — 특히 **성적 회송 채널(LTI AGS 또는 OneRoster Gradebook) 지원 여부**를 확인하고, 미지원 시 평가 결과는 바당2 기록·리포트로 제공(성적부 반영은 교사 수동)하는 분기를 둔다. 연계 불가 항목은 자체 보완(최소 명부·SSO) 분기 마련. **LTI 실행의 임베딩 환경 제약**(브라우저 서드파티 쿠키·스토리지 차단 — 전자칠판·태블릿 포함)에서의 SSO 왕복·세션 유지를 검수 시나리오로 명시하고, 필요 시 새 창 전환 등 우회 UX를 제안 요구사항에 둔다.
- AIEP(11시도/NIA)·교육부(KERIS) 트랙과의 **역할 분담·중복** 정리 — 바당2는 두 트랙이 비워둔 **'학생용 생성형 AI 접근 + 전체 로깅' 계층**임을 명확히. **AIEP 2차 구축('26.8.~'27.2.)과 바당2 개발 기간이 겹치므로**, 2차 구축으로 바당 측 연계 규격(SSO·OneRoster·LTI/API)이 바뀔 리스크를 전제하고 — 연계 규격 합의서에 **버전 고정·변경 통지 조항**을 두고 2차 구축 완료 시점에 재확인한다.
- 멀티모달 원본 **전량 보관 vs 텍스트+참조 보관** 정책(용량·비용·프라이버시 균형) — 음성·영상은 미성년의 생체·초상 정보로 텍스트보다 위험 등급이 높다(**프라이버시 최소화 우선**).
- ✅ **[해소] 멀티모달 전송 정책 · 학생 이미지 입력:** 「학생 이미지 입력을 열 것인가」는 **연다**로 확정한다. 단 **비전 비식별(§3-5)을 반드시 경유**한다(얼굴 탐지·마스킹 + 이름표·손글씨 OCR 후 텍스트 비식별 재투입 + EXIF 제거). **전면 개방-무처리 조합은 배제**한다. *근거:* 딥페이크를 막는 유일한 수단이 **얼굴 탐지**이고 그것이 곧 비전 비식별이다 — **넣지 않으면 다섯 기능(이미지 입력·pptx 내 사진·그림공방 생성/편집·라이브튜터)을 못 열면서 안전도 못 지키고, 넣으면 둘 다 된다.**
- **비전 비식별 목표 수치** — **얼굴 탐지 재현율 하한 · 오탐 상한 · 프레임 처리 지연(라이브튜터 p95)** 을 ISMP에서 **수치로 확정**한다. 평가셋은 **교육청이 제작**해 RFP에 첨부한다(텍스트 비식별과 동일 원칙). **경량 비전 가속기 대수·사양**도 여기서 산정한다(§3-4 ④).
- **음성 지연 목표 수치** — 「음성대화」의 자체 STT→비식별·안전필터→TTS 왕복 **체감 지연(p95)** 을 수치로 확정한다. *"벤더가 안 주니 저품질로 참는다"가 아니라 "우리 스택으로 동급 지연을 낸다"* 가 목표이므로, **수치 없이는 검수가 불가능**하다.
- ✅ **[해소] 한컴 라이선스 · hwp 쓰기 경로:** 「한컴 서버 SDK·변환 라이선스 확보」는 **불필요하다.** **공문서 법정 표준이 hwpx로 확정**(「행정업무 운영 및 혁신에 관한 규정」 개정 — 2026.5.12. 국무회의 의결, **2026.10. 의무화**)되어 **hwp(구 바이너리) 쓰기를 개통 범위에서 제외**했기 때문이다. hwpx는 **개방 표준(KS X 6101)이라 자체 구현**되고, hwp는 **레거시 자료 읽기 전용**이다. → **한컴 라이선스 협상 트랙 · Windows 변환 팜 · 인프라 부하 축 하나가 함께 소멸**한다(§3-4).
- **공문서 개방형 포맷 의무 준수 — RFP 요건:** 「행정업무 운영 및 혁신에 관한 규정」의 개방형 문서 의무(**hwpx** 기본 · ODF·**PDF/A** 병용)를 **RFP 필수 요건**으로 명시한다. 「문서공방」의 생성 산출물과 「한글 편집기」의 저장 결과가 **교육청 표준 공문·가정통신문 서식으로 한글에서 서식 손상 없이 열릴 것**을 검수 시나리오로 둔다. *(레거시 hwp 생성이 사후에 필요해지면 확장으로 붙인다 — 개통 필수에서는 제외.)*
- **벤더 DPA·국내 리전 — 「AI 선택」의 실질적 제약:** §3-5의 DPA 4개 조항(① 학습 미사용 ② 제로 리텐션 ③ 국외 이전·재위탁 제한 ④ 침해 통지)과 **국내 리전 제공 여부**를 벤더별로 확인한다 — **충족하지 못하는 벤더는 어댑터 등록 자체가 불가**하다. **OpenAI 등의 국내 리전·제로 리텐션 공공 조건 제공 여부가 "복수 모델 선택"의 전제**이며, 불충족 시 **단일 벤더로 시작하는 분기**를 둔다.
- 비식별 **탐지 목표(재현율)·평가셋·검증 방법**은 ISMP에서 **수치로 확정** — 수치 없는 요건은 인수 검수가 불가능하다. 평가셋은 **교육청이 제작해 RFP에 첨부**한다(업체 자체 제작 금지). **위기감지도 동일 원칙** — 평가셋(교육청 제작)·재현율 하한·오탐 상한·운영 지표(알림 건수·확인 소요)를 수치로 확정한다.
- 스트리밍 **출력 게이트**(§3-5)의 재현율·체감 지연 **목표 수치** — ISMP에서 확정(평가셋은 교육청 제작).
- 교과서 코퍼스 **라이선스 협상 트랙**(발행사·비용) — ISMP에서 타진하되 **시범 필수 경로 아님**(§3-5 권리 3층).

- **⚠️ PIA 대상성은 미확정 — 단정 서술 금지:** 학생·교직원 약 8만명대로 **시행령 §35의 50만/100만 기준을 자력 충족하지 않는다.** 대상성은 **① 5만명 이상 민감정보·고유식별정보 처리(1호) ② 기존 영향평가 대상 파일의 운용체계 변경(4호) ③ 바당(11개 시·도 공동) 연계 여부**에 따라 결정되며 **ISMP 단계에서 확정**한다. *"교육청 규모면 당연히 PIA 대상"* 이라고 쓰지 않는다. *(참고: 「개인정보 영향평가에 관한 고시」 제2025-7호(2025.9.5 시행)가 **AI 평가 분야·항목을 신설**했고, 「수행안내서」(2025.10 개정)의 명시적 목적이 **"공공기관의 AI 도입·활용 시 개인정보보호 강화"** 다 — **대상이 된다면 마스킹 해제 기능의 허용 여부는 PIA에서 판정된다.**)*
- **⏱️ 시점 민감성 — 발주·ISMP 시점 재확인 필수:** 본 조사는 **2026-07 기준**이다. **2026.9.11. 시행 예정 PIPA 개정으로 과징금 상한이 상향**되고, **AI 기본법이 2026.1.22. 시행**됐다. **ISMP·RFP 확정 시점에 법령·고시·안내서를 재확인**한다.

- 기능 모듈 우선순위·단계(MVP vs 확장)와 **확장 표준(LTI·MCP)·심사 체계** 범위 확정. **MCP는 아직 빠르게 바뀌는 규격이므로 RFP에 버전을 고정하고 변경 통지 조항**을 둔다(바당 연계 규격과 같은 취지). **QTI 3.0·CASE·LTI는 안정된 표준이라 이 걱정이 없다 — 버전 관리 대상은 MCP 하나뿐이다.**
- **⚠️ 「학생 1인당 API 사용료 산출」 재산정 — 에이전트 팬아웃 반영:** 기존 산출은 **요청 1건 = API 호출 1회**를 전제로 한다. 그러나 §3-3의 **에이전트 오케스트레이션**(교차 채점·위기감지 다중 검증·문항 검증·깊이조사)에서는 **요청 1건이 호출 수백 회**가 될 수 있다. ① 팬아웃 기능별 **예상 호출 배수**를 산정하고 ② **작업 예산 상한**을 정책 수치로 확정하며 ③ 그 결과로 **1인당 사용료·시범 운영비를 재산정**한다(기능 7). **이 전제를 손보지 않으면 운영비가 실제의 몇 배로 튄다.**

- **현행 교직원 AI 업무 서비스와의 전환 관계:** ① 바당2 개통(2028-09) 시 **대체·병존·단계적 이관** 중 무엇으로 갈 것인지 ② 그때까지 축적되는 **사용 데이터(도구·챗봇 제작 이력, 업무 유형별 사용량)를 바당2 요구사항·기본 스킬 설계의 근거로 활용**할 방안 ③ **예산 중복 구간**(구축 기간 중 현행 서비스 구독료 병행)의 처리. 상세 격차는 **[기능 분석](<참고자료/현행-교직원AI업무서비스-기능분석.md>)** 참조.
- **AIDT·KERIS 트랙의 문항·성취기준 표준 채택 여부** — §2의 바당 연계 표준에는 xAPI·LOM만 열거되어 있고 **문항(QTI)·성취기준(CASE) 표준은 언급이 없다.** AIDT가 이미 QTI·CASE를 쓴다면 채택은 자명하고, **쓰지 않는다면 제주가 먼저 쓰는 것이 오히려 이점**이다(제주 콘텐츠가 곧 표준 자산이 되어 타 시도·에듀테크로 교환·수출 가능 — 기능 10 「제주 에듀테크 산업 육성」의 전제).

- **⚠️ 사업 범위 확대에 따른 FP·기간·인프라 재산정 (최우선):** 기능 1의 「개통 필수 목록」은 **사업 범위를 실질적으로 키운다** — 코드 실행 샌드박스 · 콘텐츠뷰 · 검색 프록시 · 지식창고 · 그림공방 · 음성 스택 · 문서공방(hwp) · 비전 비식별은 각각 **독립 기능군**이며, §4의 기능 서술만으로는 FP 산정에 잡히지 않는다. §3-4의 부하 축도 **셋 → 다섯**으로 늘었다.
 - **§5의 개발 기간(T+2~T+7, 4~5개월)에 이 범위가 들어가는지는 FP를 산정해 봐야 안다.** 세 갈래가 있다 — ⓐ 개발 기간 확대(개통 2028-09-01 사수 가능성 역산) ⓑ 개통 필수를 2단계로 분할(개통 시 A군 / 6개월 내 B군) ⓒ 병렬 개발 확대(사업비 증가).
 - **순서:** **지금은 범위를 확정**하고 → **ISMP(~2026-09-30)에서 FP·기간·인프라를 산정**하고 → 그 결과로 **일정·단계를 조정**한다. **범위를 미리 줄이면 예산에 실리지 않고, 예산에 실리지 않으면 개통 후에는 만들 수 없다** — 그것이 교육용 AI 서비스가 파일럿만 하고 사라지는 경로다(기능 1 반면교사).
- **교직원 업무 활용:** 구조(업무 세션·업무 화면·정책 분리)는 **세션정책으로 확정**(§3-6). 잔여 확인 사항 둘 — ① **OneRoster sourcedId가 SAML 속성과 LTI 클레임/NRPS 응답 양쪽에 실려 오는지** NIA 연계 합의서에 명기(단일 신원의 결합 키 — 세션정책 §8) ② **교직원 로그 거버넌스 제정**(열람 주체·요건 · 근무평정·징계 목적 사용 금지 · 보관 단축 — 세션정책 §9, **시범 1단계(업무 세션 개방)의 선행 조건**).

**연계·상호운용 표준**

| 약어 | 풀이 |
| --- | --- |
| LMS | Learning Management System(학습관리시스템) — 수업·학생·학습활동을 관리하는 플랫폼. 바당이 이에 해당. |
| LRS | Learning Record Store(학습기록저장소) — 학습활동 기록(xAPI)을 표준 형식으로 모아두는 저장소. 바당이 보유. 바당2의 적재(환류) 여부는 §3-2 — 학습기록 정본은 바당2 자체 DB. |
| xAPI | Experience API — "누가·무엇을·했다" 형태로 학습 경험을 기록·전송하는 국제 표준(미국 ADL 개발·IEEE 표준화). |
| 완전학습 | Mastery Learning — 블룸(1968)이 정립한 수업 모형. 형성평가와 교정 절차의 순환으로 모든 학생이 숙달에 도달하게 한다. 기능 3 대표 시나리오(완전학습 루프)의 이론적 기반 — 「블룸 2σ 이론적 배경」 참조. |
| SSO | Single Sign-On(통합인증) — 한 계정으로 여러 시스템에 로그인. |
| SAML 2.0 | Security Assertion Markup Language — SSO를 구현하는 인증 표준. |
| OneRoster | 학생·교사·수업 명부(클래스·배정)와 **성적**을 주고받는 국제 표준(1EdTech 제정). |
| LTI | Learning Tools Interoperability — 외부 학습도구를 LMS 안에 끼워 SSO된 채 쓰게 하는 표준(1EdTech 제정 · 본 사업은 **LTI 1.3/Advantage**). |
| LOM | Learning Object Metadata — 학습자료에 붙이는 메타데이터 표준. |
| **QTI** | Question & Test Interoperability — **문항·평가(정답·해설·루브릭)를 교환하는 국제 표준**(1EdTech). 바당2는 **QTI 3.0 프로파일**(실제 사용 문항 유형만 규정한 부분집합)을 채택한다 — 자체 문항 포맷을 만들지 않는다(§3-3 · 기능 10). |
| **CASE** | Competencies and Academic Standards Exchange — **성취기준·역량 체계를 교환하는 국제 표준**(1EdTech). 기능 3 스킬 맵의 뼈대이며, 국가 교육과정 성취기준 코드를 이 위에 얹는다 — 교육과정 개정 시 **코드 매핑 교체만으로 대응**한다. |
| **Common Cartridge** | 학습 콘텐츠를 한 덩어리로 묶어 교환하는 패키징 표준(1EdTech). 확장 패키지의 콘텐츠 층(§3-3). |
| **1EdTech** | 교육 상호운용 표준 제정 기관(구 IMS Global) — LTI · QTI · CASE · Common Cartridge · OneRoster의 제정 주체. |
| MCP | Model Context Protocol — AI 모델에 외부 도구·데이터를 표준 방식으로 연결하는 프로토콜. *(사람이 도구를 쓰게 하는 LTI와 층위가 다르다 — 모델이 도구를 쓴다.)* |
| OpenAPI | REST API 명세를 기술하는 표준(연동·문서화용). |
| 어댑터 | 코어를 고치지 않고 새 모델·벤더를 끼우기 위한 **내부 확장점**(공통 인터페이스 + 모델 레지스트리 등록). 유통되는 물건이 아니다(§3-3). |
| 확장 패키지 | 마켓에서 **심사·배포·설치되는 유통 단위**. 매니페스트·시스템 프롬프트·MCP 도구 연결·(필요 시) LTI 등록·콘텐츠를 한 덩어리로 묶은 것(기능 10). |

**AI·기술**

| 약어 | 풀이 |
| --- | --- |
| LLM | Large Language Model(대규모 언어모델) — 생성형 AI의 핵심 모델(예: Claude). |
| MDM | Mobile Device Management(기기관리) — 단말을 원격 관리·통제하는 시스템. |
| API Gateway | 외부 호출을 표준·보안 경유시키는 관문. |
| PII | Personally Identifiable Information — 개인을 식별할 수 있는 정보(개인정보). |
| 가명정보 · 가명 토큰 | 실명을 대체 식별자로 치환해 처리하는 개인정보 — 매핑 보유로 재식별 가능하므로 **익명이 아니며** 안전조치 의무 대상(개인정보보호법 §28-4). 벤더 왕복 구간에는 가명 토큰만 흐른다(§3-5). |
| EXIF | 사진 파일에 자동 기록되는 촬영 메타데이터(위치·일시·기기) — 학생 이미지 입력 시 **자동 제거 대상**(§3-5 비전 비식별). |
| **비전 비식별** | 이미지·영상 경로의 **얼굴 탐지·마스킹 · 이름표·손글씨 OCR 후 텍스트 비식별 재투입 · EXIF 제거.** 텍스트 비식별로는 걸러지지 않는 축이며, **경량 비전 가속기**를 요한다(§3-5 · §3-4 ④). |
| **딥페이크** | 실제 인물의 얼굴·음성을 합성해 허위 이미지·영상을 만드는 것. 이미지 편집 모델의 **인물 일관성 유지** 능력에서 비롯하며, **얼굴 포함 이미지의 편집 도구 투입 차단**이 유일한 방어선이다(기능 5 · §3-5). |
| **샌드박스** | 코드를 **격리된 환경에서 실행**해 본체 시스템·타 사용자 데이터에 접근하지 못하게 하는 장치. 바당2는 「실행실」·「문서공방」·코드 실행형 확장이 **하나의 코어 공용 샌드박스**를 쓴다(§3-4 ⑤ · 기능 10). |
| **서버 실행형 도구 / client-side 도구** | 벤더 서버에서 실행되는 도구(코드 실행·웹 검색 등)와, 벤더가 스키마만 정의하고 **호출자가 직접 실행**하는 도구. 바당2는 **후자만 쓴다** — 전자는 데이터가 벤더 인프라로 넘어가 로깅·비식별·국내 보관이 끊긴다(§3-1 두 번째 금지선). |
| **hwpx** | 개방형 한글 문서(**KS X 6101** · XML/ZIP). **공문서 법정 표준** — 「행정업무 운영 및 혁신에 관한 규정」 개정(2026.5.12. 국무회의 의결)으로 **2026.10.부터 공공기관 기본 포맷 의무화**(ODF·PDF/A 병용). 전환 이유가 **"AI가 읽기 좋은 구조"** 이며, 바당2의 「문서공방」·「한글 편집기」·「규정·지침 질의」가 그 전제 위에 선다. **자체 구현 가능**(개방 표준). |
| **hwp** (구 바이너리) | 폐쇄형 한글 문서. 바당2는 **읽기만 지원**(축적된 레거시 자료 활용). **쓰기는 개통 범위에서 제외** — 공문서 표준이 hwpx로 전환되었기 때문이며, 이로써 한컴 서버 SDK·변환 라이선스·Windows 변환 팜이 모두 불필요해졌다(§6). |

**인프라·인증·제도**

| 약어 | 풀이 |
| --- | --- |
| CSAP | 클라우드 보안인증(Cloud Security Assurance Program) — 공공기관용 클라우드 보안 인증. |
| HA | High Availability(고가용성) — 장애 시에도 중단 없도록 이중화한 구성. |
| DR · RPO/RTO | 재해복구(Disaster Recovery) / 복구 시점 목표(RPO — 데이터 손실 허용 한도) · 복구 시간 목표(RTO — 중단 허용 시간). RFP에 수치로 명시(§3-4). |
| NTP | Network Time Protocol — 서버 시각 동기화 표준. 로그 무결성(변조 방지·시각 증빙)의 전제(§3-5). |
| KWCAG | 한국형 웹 콘텐츠 접근성 지침 — 공공 웹 접근성의 법정 기준. 준수·인증 취득을 RFP 필수 산출물로 명시(기능 6). |
| PIA | 개인정보 영향평가(Privacy Impact Assessment) — 개인정보 처리 위험을 사전 평가. |
| RFP | 제안요청서(Request for Proposal) — 사업 발주 요구사항 문서. |
| MVP | 최소기능제품(Minimum Viable Product) — 핵심 기능만 갖춘 1차 산출물. |
| ISMP | 정보시스템 마스터플랜(Information Systems Master Plan) — 단일 시스템 구축 전, 요구사항을 상세 분석하고 기능점수를 산정해 제안요청서(RFP)를 도출하는 선행 활동(바당2는 개발TF 자체 수행). |
| ISP | 정보화전략계획(Information Strategy Planning) — 기관 전체·중장기 정보화 전략·로드맵 수립(ISMP보다 광의). |
| FP | 기능점수(Function Point) — 소프트웨어 규모를 기능 단위로 측정하는 표준. 공공 SW 사업비 산정 기준. |
| SRS | 요구사항 명세서(Software Requirements Specification) — 기능·비기능 요구를 상세 기술한 문서. |
